Practice 1

Core Architecture and Fundamentals

Kafka is a distributed commit log. The durable unit is not a queue message sitting in memory; it is a record appended to a partition log, replicated to other brokers, and consumed by offset.

ProduceKey and batch

Producers serialize keys and values, batch records, compress them, and send each batch to the leader for the selected partition.

AppendLeader writes

The leader broker appends the batch to the partition log and assigns monotonically increasing offsets.

ReplicateFollowers fetch

Replica brokers pull from the leader and stay in the in-sync replica set when they keep up.

ConsumeGroups commit

Consumers fetch by offset and commit their progress to Kafka, so processing can resume after failures.

Atomic parts

  • Broker: a Kafka server process that stores partitions and handles client requests.
  • Topic: a named stream split into one or more partitions.
  • Partition: an ordered append-only log. Ordering is guaranteed only within one partition.
  • Replica: a copy of a partition. One replica is leader; the rest fetch from it.
  • Offset: the position of a record inside a partition. It is a cursor, not a timestamp.
  • Consumer group: cooperating consumers where each partition is assigned to one group member at a time.

Durability knobs

  • replication.factor=3 tolerates one broker loss while keeping a clean leader path.
  • min.insync.replicas=2 requires at least two in-sync replicas for acknowledged writes when producers use acks=all.
  • unclean.leader.election.enable=false favors consistency over availability by avoiding stale replicas as leaders.
  • retention.ms and retention.bytes delete old log segments; cleanup.policy=compact keeps the latest value per key.

Producer and consumer mechanics

Producer path

  1. Serialize key and value with matching client serializers.
  2. Choose a partition by explicit partition, key hash, or sticky batching for null keys.
  3. Accumulate records in memory until batch.size, linger.ms, or memory pressure triggers send.
  4. Send to the current partition leader. Retry safely with idempotence enabled.
  5. Receive acknowledgement based on acks and ISR state.

Consumer path

  1. Join a group with group.id and subscribe to topics.
  2. Coordinator assigns partitions to members with the selected assignor.
  3. Consumer fetches from leaders and processes records in offset order per partition.
  4. Offsets commit automatically or manually after processing.
  5. If heartbeats stop beyond session.timeout.ms, the group rebalances.
Production default: use idempotent producers, acks=all, a replication factor of at least 3 for important topics, and commit offsets only after side effects are complete.

KRaft consensus vs ZooKeeper legacy

Modern Kafka runs metadata through KRaft, Kafka's Raft-based quorum controller. ZooKeeper mode is legacy: you will still meet it in older estates, but new deployments should start with KRaft unless a vendor support matrix forces otherwise.

AreaKRaft modeZooKeeper legacy mode
MetadataStored in Kafka metadata log and managed by quorum controllers.Stored in ZooKeeper znodes; Kafka controller watches ZooKeeper state.
Controller processDedicated controllers or combined broker/controller nodes for small labs.One broker becomes active controller; ZooKeeper coordinates broker liveness.
Identitynode.id, process.roles, controller.quorum.voters.broker.id and zookeeper.connect.
Operational riskOne less distributed system to run; controller quorum still needs odd-sized, stable voters.Kafka availability depends on ZooKeeper health and session behavior.
MigrationPlan feature level, controllers, rollback windows, client compatibility, and policy plugins.Existing clusters may need staged migration before ZooKeeper removal.

Setup, Orchestration, and K8s

Kafka setup is mostly about making identity, listeners, storage, and replication boring. Most incidents start with advertised listener mistakes, under-sized disks, missing rack awareness, or accidental single-replica topics.

KRaft broker/controller properties

Use separate controllers for production. A combined node is acceptable for local labs or tiny non-critical environments.

properties - config/kraft/server.properties for one lab node
process.roles=broker,controller
node.id=1
controller.quorum.voters=1@localhost:9093

listeners=PLAINTEXT://:9092,CONTROLLER://:9093
advertised.listeners=PLAINTEXT://localhost:9092
controller.listener.names=CONTROLLER
listener.security.protocol.map=CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT
inter.broker.listener.name=PLAINTEXT

log.dirs=/var/lib/kafka/data

Bare metal and VM checklist

  • Run brokers on stable local SSD or high-IOPS block storage. Kafka prefers predictable disks over clever shared filesystems.
  • Use stable DNS names and never expose internal broker names to external clients unless routing supports them.
  • Set rack or zone metadata with broker.rack so replicas spread across failure domains.
  • Separate data logs from OS logs; alert when disk approaches 70-80 percent.
  • Pin JVM heap conservatively. Page cache is part of Kafka performance; do not starve the OS.

Day-to-Day CLI Operations and Examples

The Kafka shell tools are still the fastest way to prove cluster state. Set a bootstrap variable once, then keep commands repeatable and explicit.

bash - operator shell setup
$ export BOOTSTRAP="localhost:9092"
$ export TOPIC="orders.v1"
$ export GROUP="orders-api"
$ alias kt='bin/kafka-topics.sh --bootstrap-server "$BOOTSTRAP"'
$ alias kc='bin/kafka-configs.sh --bootstrap-server "$BOOTSTRAP"'
$ alias kg='bin/kafka-consumer-groups.sh --bootstrap-server "$BOOTSTRAP"'

Topics

bash - create, describe, alter, and list topics
$ bin/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" \
  --create --topic "$TOPIC" \
  --partitions 12 \
  --replication-factor 3 \
  --config min.insync.replicas=2 \
  --config retention.ms=604800000

$ bin/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --describe --topic "$TOPIC"
$ bin/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --alter --topic "$TOPIC" --partitions 24
$ bin/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --list

Topic config changes

bash - inspect and change topic configuration
$ bin/kafka-configs.sh --bootstrap-server "$BOOTSTRAP" \
  --entity-type topics --entity-name "$TOPIC" --describe

$ bin/kafka-configs.sh --bootstrap-server "$BOOTSTRAP" \
  --entity-type topics --entity-name "$TOPIC" --alter \
  --add-config retention.ms=1209600000,segment.bytes=536870912

$ bin/kafka-configs.sh --bootstrap-server "$BOOTSTRAP" \
  --entity-type topics --entity-name "$TOPIC" --alter \
  --delete-config segment.bytes
Destructive command: --delete --topic removes data when topic deletion is enabled. Verify the exact topic, backups, consumers, and environment before running it.

bash - dry-run then execute an offset reset
# Stop all consumers in the group before resetting offsets.
$ bin/kafka-consumer-groups.sh --bootstrap-server "$BOOTSTRAP" \

  --topic "$TOPIC" \

Practice 4

Data Integration and Schemas

Kafka becomes a platform when source systems, sink systems, and event contracts are managed deliberately. Connect moves data. Schema Registry makes data evolution observable and enforceable. Streams and ksqlDB transform the data in motion.

Kafka Connect

Architecture

  • Worker: JVM process running standalone or distributed mode.
  • Connector: logical job configuration, such as JDBC source or S3 sink.
  • Task: parallel execution unit assigned to workers.
  • Converter: maps Connect's internal data model to Kafka bytes, often JSON, Avro, Protobuf, or JSON Schema.
  • SMT: single message transform for lightweight per-record changes.
  • Internal topics: configs, offsets, and status. Replicate them like production metadata.

Confluent Schema Registry

Schema Registry stores schemas by subject, assigns immutable IDs, checks compatibility, and lets serializers put a schema ID in each record so consumers can resolve the writer schema at read time.

CompatibilityMeaningCommon use
BACKWARDNew consumers can read data written with the previous schema.Default for many event streams where consumers deploy after producers.
FORWARDOld consumers can read data written with the new schema.Producer-first rollouts with conservative consumers.
FULLBackward and forward compatible for the last version.Shared events with many independently deployed services.
*_TRANSITIVECompatibility checked against all previous versions, not just the last.Long-lived topics and strict data contracts.

Kafka Streams and ksqlDB overview

Kafka Streams

Use Kafka Streams when transformations belong in an application artifact with code review, tests, custom logic, and application-level deployment controls.

  • Tasks map to input partitions and run inside your app instances.
  • State stores use local RocksDB and changelog topics for recovery.
  • Repartition topics appear when keying changes before joins or aggregations.
  • Monitor processing rate, commit latency, record lateness, and state restore time.

ksqlDB

Use ksqlDB when SQL-speed iteration matters: materialized views, stream-table joins, continuous queries, and quick operational exploration.

bash - ksqlDB CLI session
$ ksql http://ksqldb:8088
ksql> SHOW TOPICS;
ksql> CREATE STREAM orders_raw (order_id STRING, status STRING, total DOUBLE)
      WITH (KAFKA_TOPIC='orders.v1', VALUE_FORMAT='JSON');
ksql> CREATE TABLE order_totals AS
      SELECT status, COUNT(*) AS count, SUM(total) AS total
      FROM orders_raw GROUP BY status EMIT CHANGES;

Troubleshooting runbooks

Under-replicated partitions

  1. Run kafka-topics.sh --describe and identify partitions where ISR is smaller than replicas.
  2. Check broker logs for fetcher, disk, network, and authorization errors.
  3. Verify disk fullness and IO latency on follower brokers.
  4. Avoid broker restarts until you know whether replicas are catching up or falling further behind.
bash - identify ISR shrinkage
$ bin/kafka-topics.sh --bootstrap-server "$BOOTSTRAP" --describe \
  | awk '/Isr:/ && $0 !~ /Replicas: ([^ ]+) Isr: \1/ { print }'

Consumer lag spike

  1. Check if lag is all partitions or a single hot partition.
  2. Look for repeated rebalances in consumer logs.
  3. Compare processing time with max.poll.interval.ms.
  4. Scale consumers only up to partition count; beyond that, instances idle.
bash - lag detail by partition
$ bin/kafka-consumer-groups.sh --bootstrap-server "$BOOTSTRAP" \
  --describe --group "$GROUP" \
  | sort -k6 -nr | head -20

Partition rebalance loop

  • Increase max.poll.interval.ms if processing batches legitimately takes longer than the default.
  • Decrease batch size or work per poll if processing is too chunky.
  • Use static membership with group.instance.id for stable stateful consumers.
  • Prefer cooperative assignors where client versions support them.

Connect task failed

  • Read /connectors/<name>/status; task trace often points to serialization, auth, DB, or sink errors.
  • Restart only failed tasks first.
  • Check dead letter queue topic if configured.
  • Confirm connector plugin versions match worker and Kafka client compatibility.
bash - failed task trace
$ curl -s http://connect:8083/connectors/jdbc-orders-source/status | jq '.tasks[] | select(.state!="RUNNING")'
$ curl -s -X POST 'http://connect:8083/connectors/jdbc-orders-source/restart?includeTasks=true&onlyFailed=true'
