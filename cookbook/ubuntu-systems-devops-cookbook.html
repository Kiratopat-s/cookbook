Practice 1

Architecture, Boot Process & systemd

Treat Ubuntu as a stack of contracts. Firmware starts a bootloader, the bootloader loads the kernel and initramfs, the kernel mounts the first root filesystem, and systemd becomes PID 1 to converge the machine toward a target.

Linux Kernel vs. User Space

The kernel is the privileged broker for CPU scheduling, virtual memory, block I/O, network I/O, device drivers, cgroups, namespaces, filesystems, security modules, and system calls. User space is everything else: shells, daemons, package tools, journald, sshd, container runtimes, editors, and application processes.

Kernel mode

Runs with hardware privilege. It maps memory, schedules tasks, receives interrupts, mediates syscalls, and enforces capability checks.

User mode

Runs processes with restricted privileges. Programs ask the kernel for work via syscalls such as open, read, mount, and clone.

Init boundary

The first user-space process, PID 1, is special. On Ubuntu it is normally systemd, responsible for service supervision and shutdown ordering.

Virtual memory gives each process a private address space. Page tables translate virtual addresses to physical frames. The kernel can overcommit memory, reclaim page cache, swap cold pages, and kill a process when memory pressure cannot be resolved. The scheduler assigns runnable tasks to CPUs based on policy, priority, cgroup weights, CPU affinity, and load balancing. Block devices are exposed under /dev; filesystems translate path operations into block I/O; VFS lets ext4, xfs, zfs, tmpfs, procfs, and sysfs present a common interface.

Boot Process: Firmware -> GRUB -> Kernel -> Initramfs -> systemd

Ubuntu boot failures are easiest to debug when you know the handoff points. Firmware finds a boot entry, GRUB reads its configuration, the kernel receives boot arguments, the initramfs provides early user space and storage/network drivers, and systemd takes over once the real root filesystem is mounted.

StagePrimary filesWhat can breakDiagnostic entry point
UEFI/BIOS/boot/efi, firmware NVRAMMissing boot entry, bad disk order, Secure Boot mismatchefibootmgr -v
GRUB/boot/grub/grub.cfg, /etc/default/grubWrong root UUID, old kernel menu, invalid boot argsGRUB menu, recovery mode
Kernel/boot/vmlinuz-*Driver panic, unsupported hardware, bad kernel parameterjournalctl -k -b -1
Initramfs/boot/initrd.img-*, /etc/initramfs-tools/Cannot unlock disk, cannot assemble LVM/RAID, missing moduleBusyBox initramfs shell
systemd/etc/systemd/system, /lib/systemd/systemFailed mounts, dependency cycles, service timeoutsystemctl --failed
Safety

Changing bootloader, initramfs, disk UUID, LUKS, or /etc/fstab configuration can make a host unbootable. Keep console access, a rescue ISO, a tested snapshot, or a cloud provider serial console before rebooting production systems.

Keep boot configuration declarative where possible. Edit /etc/default/grub, regenerate GRUB, regenerate initramfs after early-boot module or crypttab changes, and verify the menu before reboot.

# Show installed kernels
dpkg -l 'linux-image*' | awk '/^ii/{print $2, $3}'

# Edit kernel command-line defaults, then regenerate GRUB
sudoedit /etc/default/grub
sudo update-grub

# Regenerate initramfs for all installed kernels
sudo update-initramfs -u -k all

# Confirm boot entries on UEFI systems
sudo efibootmgr -v
GRUB_DEFAULT=0
GRUB_TIMEOUT_STYLE=menu
GRUB_TIMEOUT=5
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"
GRUB_CMDLINE_LINUX="systemd.unified_cgroup_hierarchy=1"
  1. If the machine does not reach GRUB, check firmware boot order, disk visibility, Secure Boot state, and UEFI entries.
  2. If GRUB loads but the kernel panics, boot an older kernel from Advanced options.
  3. If initramfs drops to BusyBox, run blkid, ls /dev/mapper, lvm vgchange -ay, and confirm root UUIDs.
  4. If systemd emergency mode appears, run journalctl -xb, fix failed mounts or units, then systemctl default.
journalctl -xb --no-pager | tail -200
systemctl --failed
findmnt --verify --verbose
mount -o remount,rw /
sudoedit /etc/fstab
systemctl daemon-reload
systemctl default

systemd Units, Targets & Service Management

systemd models the host as a dependency graph of units. Service units start daemons, socket units activate services on demand, timer units replace many cron jobs, mount units track filesystems, and targets group the desired boot state. Parallel startup works because units declare ordering and requirement relationships instead of a single serial boot script.

Ordering vs. requirement

After=network-online.target controls order. Wants=network-online.target or Requires= controls whether another unit is pulled into the transaction.

Unit locations

Vendor units live in /lib/systemd/system. Local overrides live in /etc/systemd/system. Runtime units live in /run/systemd/system.

[Unit]
Description=Example API service
Documentation=https://example.internal/runbooks/example-api
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=example
Group=example
WorkingDirectory=/opt/example-api
EnvironmentFile=-/etc/default/example-api
ExecStart=/usr/local/bin/example-api --config /etc/example-api/config.yaml
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5s
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=full
ProtectHome=true
ReadWritePaths=/var/lib/example-api /var/log/example-api

[Install]
WantedBy=multi-user.target
# Service lifecycle
sudo systemctl start nginx
sudo systemctl stop nginx
sudo systemctl restart nginx
sudo systemctl reload nginx
sudo systemctl enable --now nginx
sudo systemctl disable --now nginx

# Status and dependency graph

Practice 2

Package Management & Software Lifecycles

Ubuntu software management has layers: dpkg installs local Debian packages, apt resolves dependencies from repositories, Snap provides confined app bundles, and PPAs add additional apt sources. Production systems need pinning, provenance, staged upgrades, and rollback plans.

apt and dpkg Under the Hood

apt is the operator-friendly dependency resolver. It downloads package indexes from configured repositories, calculates dependency transactions, fetches .deb files, and calls dpkg. dpkg unpacks archives, runs maintainer scripts, records ownership in /var/lib/dpkg/status, and leaves package states such as installed, half-configured, or config-files.

Important state

/etc/apt/sources.list, /etc/apt/sources.list.d/, /var/lib/apt/lists/, /var/cache/apt/archives/, /var/lib/dpkg/.

Important logs

/var/log/apt/history.log, /var/log/apt/term.log, /var/log/dpkg.log, and journal entries for services restarted by packages.

Practice

In production, prefer apt-get in scripts because its output and prompts are more stable. Use apt interactively for human workflows.

# Refresh repository metadata
sudo apt update

# Show available upgrades without installing
apt list --upgradable
sudo apt-get --just-print upgrade

# Install, remove, purge
sudo apt install nginx
sudo apt remove nginx
sudo apt purge nginx
sudo apt autoremove --purge

# Package discovery and provenance
apt search 'prometheus node exporter'
apt show prometheus-node-exporter
apt policy openssh-server
dpkg -L openssh-server
dpkg -S /usr/sbin/sshd

# Repair interrupted transactions
sudo dpkg --configure -a
sudo apt -f install

# Hold a risky package during a change freeze
sudo apt-mark hold linux-generic
apt-mark showhold
sudo apt-mark unhold linux-generic

Runbook

  1. Check locks with ps aux | grep -E 'apt|dpkg'. Do not remove locks while a package tool is still running.
  2. Read /var/log/apt/term.log for maintainer script failure output.
  3. Run sudo dpkg --configure -a to finish interrupted package configuration.
  4. Run sudo apt -f install to resolve broken dependencies.
  5. Use apt policy <package> to detect repository priority problems.
sudo tail -200 /var/log/apt/term.log
sudo tail -100 /var/log/dpkg.log
grep -R --line-number 'NO_PUBKEY\\|EXPKEYSIG\\|BADSIG' /var/log/apt /var/log/syslog 2>/dev/null
sudo apt update -o Debug::Acquire::https=true

Snap, PPAs & Software Supply Chain Choices

Snap packages are mounted SquashFS application bundles managed by snapd. They can use confinement profiles, interfaces, channels, and automatic refreshes. PPAs are Launchpad-hosted apt repositories, useful for newer packages but risky when they override base libraries. Decide per workload whether you want Ubuntu archive stability, vendor apt repositories, Snap confinement, or source-built artifacts.

SourceBest forRiskControl
Ubuntu archiveBase OS, security updates, stable servicesVersions may be olderHigh with apt pinning and mirrors
Vendor apt repoAgents, databases, runtimes with vendor supportKey and repository trust boundaryHigh if pinned and scoped
PPADeveloper workstations, narrow package needsCan replace dependencies globallyMedium; pin aggressively
SnapApps with bundled dependencies and confinementRefresh cadence and snapd dependencyMedium; channels and holds
Source buildCustom patches, isolated appliancesPatch tracking and reproducibilityHigh if built in CI with SBOM and signatures
 
snap list
snap info lxd
sudo snap install lxd --channel=5.21/stable
sudo snap refresh --list
sudo snap refresh lxd --channel=5.21/stable
snap connections lxd
sudo snap connect <snap>:<plug> <snap>:<slot>
sudo snap set system refresh.hold="$(date --date='+7 days' --iso-8601=seconds)"
# Add only when you trust the maintainer and understand dependency impact
sudo add-apt-repository ppa:<owner>/<ppa-name>
sudo apt update
apt policy <package>

# Remove PPA packages by downgrading to Ubuntu archive when possible
sudo apt install ppa-purge
sudo ppa-purge ppa:<owner>/<ppa-name>
  1. Mirror or proxy package sources where possible.
  2. Promote updates through dev, staging, canary, and production rings.
  3. Snapshot VM disks or volumes before kernel, libc, database, and storage driver upgrades.
  4. Record apt-mark showmanual, dpkg -l, and service status before large changes.
  5. Use unattended upgrades for security patches only when alerting, reboot policy, and rollback are defined.
mkdir -p /var/tmp/preupgrade-$(date +%F)
cd /var/tmp/preupgrade-$(date +%F)
uname -a > uname.txt

apt-mark showmanual > apt-manual.txt
systemctl --failed --no-pager > systemd-failed.txt

User, Security & Access Management

Ubuntu security starts with Unix identity, file permissions, sudo boundaries, SSH policy, kernel security modules, firewall rules, patching, and logs that prove what happened.

Users, Groups, POSIX Permissions & ACLs

Linux authorization begins with user IDs, group IDs, mode bits, and capabilities. A process carries a real/effective UID and GID set; the kernel checks ownership, mode bits, ACL entries, mount options, Linux capabilities, and security module policy before allowing operations.

Sticky bitOnly owner can remove entries in shared dirchmod 1777 /shared/tmpCommon on /tmp
ACLExtra allow entries beyond mode bitsgetfacl fileCan surprise scripts that only inspect ls -l
 
# User and group facts
id
id <user>
getent passwd <user>
getent group sudo

# Create a service account without an interactive login
sudo adduser --system --group --home /var/lib/example --shell /usr/sbin/nologin example

# Permissions and ACLs
stat -c '%A %a %U:%G %n' /etc/ssh/sshd_config

sudo chmod 0644 /etc/ssh/sshd_config
getfacl /srv/app
sudo setfacl -m u:deploy:rwx /srv/app
sudo setfacl -m d:u:deploy:rwx /srv/app
Safety

A recursive chmod or chown from the wrong directory can break the OS. Always run pwd, use absolute paths, and test with find ... -print before adding -exec.

Most SSH failures are file permission problems, wrong key, disabled account, AllowUsers mismatch, expired account, PAM policy, or a network path/firewall issue. Let ssh -vvv tell you where negotiation stops.

AppArmor Profiles

AppArmor is a Linux security module that confines programs by path-based policy. Profiles can allow, deny, audit, or complain about file, capability, network, and mount actions. Ubuntu ships many profiles, and operators commonly write profiles for custom daemons exposed to untrusted input.

#include <tunables/global>

/usr/local/bin/example-api {
  #include <abstractions/base>
  #include <abstractions/nameservice>

  capability net_bind_service,
  network inet stream,

  /usr/local/bin/example-api r,
  /etc/example-api/** r,
  /var/lib/example-api/** rwk,
  /var/log/example-api/** rw,
  /tmp/** rw,

  deny /etc/shadow r,
}
sudo aa-status
sudo apparmor_parser -r /etc/apparmor.d/usr.local.bin.example-api
sudo aa-complain /etc/apparmor.d/usr.local.bin.example-api
sudo aa-enforce /etc/apparmor.d/usr.local.bin.example-api
sudo aa-logprof
sudo journalctl -k -g 'apparmor="DENIED"'
  1. Confirm the profile is loaded and enforced with aa-status.
  2. Read kernel audit lines with journalctl -k -g 'apparmor="DENIED"'.
  3. Temporarily switch to complain mode to collect intended access without blocking.
  4. Use aa-logprof to review candidate rules, then manually simplify them.
  5. Reload the profile and restart the confined service.
Practice 4

Networking & Firewall Configuration

Ubuntu networking is a contract between declarative configuration, kernel interfaces, routes, resolver state, and packet filtering. Netplan writes backend config, iproute2 shows truth in the kernel, systemd-resolved manages DNS views, and UFW simplifies firewall policy.

Netplan and iproute2

Netplan reads YAML under /etc/netplan/, validates it, and generates configuration for a renderer such as systemd-networkd on servers or NetworkManager on desktops. The kernel only knows interfaces, addresses, routes, rules, neighbors, and sockets; iproute2 is the authoritative CLI for that state.

network:
  version: 2
  renderer: networkd
  ethernets:
    ens160:
      dhcp4: false
      addresses:
        - 10.20.30.15/24
      routes:
        - to: default
          via: 10.20.30.1
      nameservers:
        search:
          - prod.example.com
        addresses:
          - 10.20.1.53
          - 10.20.1.54
network:
  version: 2
  renderer: networkd
  ethernets:
    eno1: {}
    eno2: {}
  bonds:
    bond0:
      interfaces:
        - eno1
        - eno2
      parameters:
        mode: 802.3ad
        lacp-rate: fast
        mii-monitor-interval: 100
  vlans:
    bond0.120:
      id: 120
      link: bond0
      addresses:
        - 172.20.120.10/24
      routes:
        - to: default
          via: 172.20.120.1
Safety

Use netplan try over remote SSH. It rolls back if you do not confirm. A bad default route or interface name can strand a host until console access.

sudo netplan status --all

systemd-resolved and DNS Management

systemd-resolved provides local DNS resolution, split DNS, DNSSEC settings, cache, and the stub resolver commonly found at 127.0.0.53. Applications normally call libc resolver functions, which read /etc/resolv.conf; on Ubuntu this often points to the local stub managed by resolved.

resolvectl status
resolvectl dns
resolvectl domain
resolvectl query ubuntu.com
resolvectl query internal.service.prod.example.com
resolvectl flush-caches
readlink -f /etc/resolv.conf
cat /etc/resolv.conf
# Confirm resolver path
getent hosts <name>
resolvectl query <name>
dig @<dns-server-ip> <name>

# Check split DNS and per-link servers
resolvectl status
networkctl status <interface>

# Read resolver logs
sudo journalctl -u systemd-resolved -b --no-pager

# Reset cache after changing records or search domains
sudo resolvectl flush-caches
sudo systemctl restart systemd-resolved

Common causes include wrong search domain, VPN split DNS priority, stale cache, /etc/resolv.conf overwritten by tooling, DNS blocked by firewall, or an application bypassing libc and using its own resolver.

UFW Firewall Configuration

UFW is a friendly frontend for host firewall policy. On modern Ubuntu it ultimately programs nftables through the distribution firewall backend. Use UFW for simple server ingress/egress policy, and drop to nftables only when you need complex sets, marks, NAT, or advanced packet classification.

# Always allow your current management path before enabling UFW.
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow OpenSSH
sudo ufw allow from <admin-cidr> to any port 22 proto tcp
sudo ufw allow 443/tcp
sudo ufw enable
sudo ufw status verbose
sudo ufw app list
sudo ufw app info OpenSSH
sudo ufw status numbered
sudo ufw delete <rule-number>
sudo ufw allow from 10.0.0.0/8 to any port 9100 proto tcp comment 'node exporter from private network'
sudo ufw deny from <bad-ip> comment 'temporary block'
sudo ufw logging medium
sudo journalctl -k -g 'UFW BLOCK'
  • Confirm the service is listening on the expected address with ss -tulpn.
  • Confirm routing path with ip route get <client-ip>.
  • Check UFW order with ufw status numbered; first match wins in many firewall chains.
  • Check kernel logs for drops with journalctl -k -g 'UFW BLOCK'.
  • For container hosts, inspect Docker or Kubernetes rules before assuming UFW owns the full path.

Storage, LVM & Filesystem Management

Storage changes are high blast-radius operations. Understand block devices, partitions, filesystems, mount units, LVM layering, and backup state before resizing or deleting anything.

Block Devices, Partitioning, ext4 and ZFS Concepts

The kernel exposes disks as block devices such as /dev/sda, /dev/nvme0n1, and virtual devices under device mapper. A partition table slices a disk, a filesystem organizes bytes into files and metadata, and a mount attaches that filesystem into the namespace.

ext4

Default general-purpose Linux filesystem with journaling, mature repair tools, online grow support, and predictable operational behavior.

Combines volume manager and filesystem concepts: pools, datasets, snapshots, checksums, compression, quotas, and replication. It requires deliberate memory and operational planning.

Logical Volume Management (LVM)

LVM adds a flexible mapping layer between disks and filesystems. Physical volumes join volume groups, volume groups allocate extents, and logical volumes become block devices that filesystems can use. This lets you grow filesystems, migrate data, snapshot volumes, and aggregate disks without changing application paths.

# Inspect first. Replace device paths deliberately.
lsblk -f
sudo pvcreate /dev/sdb
sudo vgcreate vg_data /dev/sdb
sudo lvcreate -n lv_apps -L 100G vg_data
sudo mkfs.ext4 -L apps /dev/vg_data/lv_apps
sudo mkdir -p /srv/apps
sudo mount /dev/vg_data/lv_apps /srv/apps
df -hT /srv/apps
# Add a new disk to the volume group
sudo pvcreate /dev/sdc
sudo vgextend vg_data /dev/sdc

# Grow the logical volume and filesystem together
sudo lvextend -r -L +50G /dev/vg_data/lv_apps

# Confirm result
lvs
df -hT /srv/apps

LVM snapshots are copy-on-write block devices. They are useful for short-lived backup consistency or pre-change rollback, not as long-term backups. If the snapshot fills, it becomes invalid.

sudo lvcreate -s -n lv_apps_snap -L 10G /dev/vg_data/lv_apps
sudo mkdir -p /mnt/apps-snap
sudo mount -o ro /dev/vg_data/lv_apps_snap /mnt/apps-snap
rsync -aHAX --numeric-ids /mnt/apps-snap/ backup:/backups/apps/
sudo umount /mnt/apps-snap
sudo lvremove /dev/vg_data/lv_apps_snap
sudo pvs -o+pv_used
sudo pvmove /dev/sdb /dev/sdc
sudo vgreduce vg_data /dev/sdb
sudo pvremove /dev/sdb
sudo pvs -a -o+devices
sudo vgs -o+vg_free
sudo lvs -a -o+devices,seg_monitor
sudo vgscan
sudo vgchange -ay
sudo lvdisplay -m /dev/vg_data/lv_apps
sudo journalctl -b -g 'device-mapper|LVM|dm-' --no-pager

If volumes do not appear after boot, check initramfs contents, missing disks, changed device names, LUKS unlock order, and whether the volume group was activated.

/etc/fstab and Mount Reliability

/etc/fstab declares filesystems that should mount at boot. systemd translates fstab entries into mount units. Bad fstab entries are a classic emergency-mode cause, especially when a non-critical external disk is configured without nofail or a timeout.

# <device>                                      <mountpoint> <type> <options>                         <dump> <pass>
UUID=11111111-2222-3333-4444-555555555555      /             ext4   defaults,errors=remount-ro        0      1
UUID=aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee      /srv/apps     ext4   defaults,noatime                  0      2
server.example.com:/exports/shared             /mnt/shared   nfs4   rw,nofail,x-systemd.automount,_netdev,x-systemd.device-timeout=10s 0 0
tmpfs                                          /run/app      tmpfs  size=256m,mode=0750,uid=app,gid=app 0 0
sudo findmnt --verify --verbose
sudo systemctl daemon-reload
sudo mount -a
findmnt /srv/apps
systemctl status srv-apps.mount --no-pager
Danger

Do not run fsck against a mounted read-write filesystem. Use recovery mode, a live ISO, or an unmounted secondary volume.

findmnt /srv/apps
sudo umount /srv/apps
sudo e2fsck -f /dev/vg_data/lv_apps
sudo mount /srv/apps
df -hT /srv/apps

Observability, Logging & Performance Tuning

Linux observability means correlating logs, unit state, process state, resource pressure, kernel messages, metrics, and workload expectations. Start broad, then drill into the saturated resource.

journald, journalctl and rsyslog

systemd-journald collects structured logs from the kernel, stdout/stderr of services, syslog sockets, and audit sources. journalctl queries by unit, boot, priority, time, executable, PID, and fields. rsyslog can persist classic text logs under /var/log and forward logs to central systems.

Journal paths

Runtime logs: /run/log/journal. Persistent logs: /var/log/journal. Config: /etc/systemd/journald.conf.

Classic logs

Auth: /var/log/auth.log. Syslog: /var/log/syslog. Kernel: /var/log/kern.log. Package logs: /var/log/apt/ and /var/log/dpkg.log.

# Current boot and previous boot
journalctl -b --no-pager
journalctl -b -1 --no-pager

# Unit and time filters
journalctl -u ssh -b --since '2 hours ago'
journalctl -u nginx -p warning..alert

# Kernel and pattern filters
journalctl -k -b
journalctl -g 'Out of memory|segfault|I/O error'

# Structured output for automation
journalctl -u example-api --since today -o json-pretty

# Disk usage and retention
journalctl --disk-usage
sudo journalctl --vacuum-time=14d

Prometheus Node Exporter

Node Exporter exposes Linux host metrics as HTTP text for Prometheus. It reads kernel and userspace sources such as /proc, /sys, filesystem stats, CPU, memory, disk, network, pressure stall information, and systemd collector data when enabled.

sudo apt update
sudo apt install prometheus-node-exporter
systemctl status prometheus-node-exporter --no-pager
curl -s http://127.0.0.1:9100/metrics | head
[Unit]
Description=Prometheus Node Exporter
Wants=network-online.target
After=network-online.target

[Service]
User=node_exporter
Group=node_exporter
Type=simple
ExecStart=/usr/local/bin/node_exporter \
  --web.listen-address=:9100 \
  --collector.systemd \
  --collector.processes
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=true
PrivateTmp=true

[Install]
WantedBy=multi-user.target
Safety

Node Exporter should not be exposed to the public Internet. Restrict port 9100/tcp to Prometheus, a private network, or a mutually authenticated proxy.

sudo ufw allow from <prometheus-ip> to any port 9100 proto tcp
sudo ufw status numbered
curl -s http://127.0.0.1:9100/metrics | grep '^node_memory_MemAvailable_bytes' | head
  • Host down: Prometheus scrape failure for more than a few minutes.
  • Disk full: filesystem free below threshold and predicted to fill soon.
  • Memory pressure: sustained PSI or swap churn, not just low free memory.
  • CPU saturation: high load per CPU and sustained run queue.
  • Disk latency: high await or node disk IO time.
  • Systemd failed units: failed service count greater than zero.

The Daily CLI Masterclass

Production Linux operators live in text streams. Master command composition, process control, file discovery, JSON/YAML inspection, shell safety, and bootstrapping automation.

Text Manipulation, Search and File Discovery

Prefer small, composable tools. Use rg when available for recursive search, find for metadata and actions, awk for column logic, sed for stream edits, and jq/yq for structured data instead of brittle regular expressions.

# Fast recursive search
rg -n 'PermitRootLogin|PasswordAuthentication' /etc/ssh
rg -n --hidden --glob '!/.git' 'secret|token|password' .

# Find files by metadata
find /var/log -type f -mtime -1 -size +10M -print
find /srv/app -type f -name '*.log' -print0 | xargs -0 du -h | sort -h

# Extract useful columns
ps -eo user,comm,pcpu,pmem --sort=-pcpu | awk 'NR==1 || $3 > 5'
ss -tulpn | awk 'NR==1 || /:443/'

# Safe preview before destructive action
find /tmp -type f -mtime +7 -print
# then, only after review:
find /tmp -type f -mtime +7 -delete
# JSON logs
journalctl -u example-api -o json --since '10 minutes ago' \
  | jq -r '[.__REALTIME_TIMESTAMP, .PRIORITY, .MESSAGE] | @tsv'

# YAML via yq when installed
yq '.network.ethernets' /etc/netplan/*.yaml

# Compare config snapshots
diff -u /etc/ssh/sshd_config.bak /etc/ssh/sshd_config

Process Control, Jobs and Signals

A process has an identity, parent, environment, file descriptors, signal handlers, namespaces, cgroup, and resource limits. Use systemd for services, shell job control for interactive work, and signals when you understand what the process does with them.

# Interactive job control
long-running-command
# Press Ctrl-Z to stop it
bg %1
jobs -l
fg %1

# Signals
kill -TERM <pid>
kill -HUP <pid>
kill -KILL <pid>   # last resort; process cannot clean up

# Process tree and environment
pstree -aps <pid>
tr '\0' '\n' < /proc/<pid>/environ
ls -l /proc/<pid>/fd
cat /proc/<pid>/limits
Rule

Use systemctl restart <unit> for services managed by systemd. Killing the child process directly can fight the supervisor, hide restart loops, or bypass ExecStop hooks.

Bash Scripting Primitives

Bash is excellent glue for host operations when scripts are small, explicit, and defensive. Use strict modes carefully, quote variables, validate inputs, log intent, trap cleanup, and make destructive behavior opt-in.

#!/usr/bin/env bash
set -Eeuo pipefail
IFS=$'\n\t'

usage() {
  printf 'Usage: %s --namespace <name> --image <image>\n' "${0##*/}" >&2
}

log() {
  printf '[%s] %s\n' "$(date --iso-8601=seconds)" "$*" >&2
}

cleanup() {
  rm -f "${tmpfile:-}"
}
trap cleanup EXIT

namespace=""
image=""
while [[ $# -gt 0 ]]; do
  case "$1" in
    --namespace) namespace="${2:-}"; shift 2 ;;
    --image) image="${2:-}"; shift 2 ;;
    -h|--help) usage; exit 0 ;;
    *) usage; exit 2 ;;
  esac
done

[[ -n "$namespace" && -n "$image" ]] || { usage; exit 2; }

tmpfile="$(mktemp)"
log "would operate on namespace=$namespace image=$image"
PatternUse it forWarning
set -eExit on unhandled failureCan surprise inside conditionals and pipelines; test scripts.
set -uCatch unset variablesUse ${var:-} for optional values.
trapCleanup temp files and locksKeep cleanup idempotent.
mktempSafe temporary filesNever invent predictable temp paths.
shellcheckStatic analysisUse comments only when intentionally ignoring a rule.

cloud-init Bootstrapping

cloud-init is early-boot automation for cloud images. It reads datasource metadata and user-data, configures identity, networking, packages, files, commands, and final scripts, then records module state so most actions run only once per instance unless cleaned.

#cloud-config
hostname: app-01
manage_etc_hosts: true
package_update: true
package_upgrade: false
packages:
  - curl
  - jq
  - prometheus-node-exporter

users:
  - default
  - name: deploy
    groups: [sudo]
    shell: /bin/bash
    sudo: ['ALL=(ALL) NOPASSWD:/bin/systemctl restart example-api,/bin/systemctl status example-api']
    ssh_authorized_keys:
      - ssh-ed25519 AAAA_REPLACE_WITH_PUBLIC_KEY deploy@example

write_files:
  - path: /etc/motd
    permissions: '0644'
    owner: root:root
    content: |
      Managed Ubuntu host. Unauthorized access is prohibited.

runcmd:
  - [ systemctl, enable, --now, prometheus-node-exporter ]
  - [ ufw, allow, from, 10.0.0.0/8, to, any, port, '9100', proto, tcp ]
  - [ ufw, --force, enable ]
cloud-init status --long
sudo cloud-init analyze show
sudo cloud-init analyze blame
sudo tail -200 /var/log/cloud-init.log
sudo tail -200 /var/log/cloud-init-output.log
sudo cloud-init query --all | less
status: done
boot_status_code: enabled-by-generator
last_update: Sun, 05 Jul 2026 02:30:18 +0000
detail:
Cloud-init completed all stages successfully.
# Do not run clean casually on production instances; it resets cloud-init state.
sudo cloud-init clean --logs
sudo reboot
  1. Validate YAML indentation and ensure the file starts with #cloud-config.
  2. Check datasource detection in /run/cloud-init/ and cloud-init query --all.
  3. Review /var/log/cloud-init.log for module errors and cloud-init-output.log for command output.
  4. Avoid long-running daemons in runcmd; install systemd units and enable them instead.
  5. Make scripts idempotent because image rebuilds and retries happen during platform automation.

Daily Incident Flow: From Symptom to Layer

When a host is unhealthy, avoid random command thrashing. Establish scope, identify the failing layer, collect current evidence, make the smallest reversible change, and record the result.

date --iso-8601=seconds
hostnamectl
uptime
systemctl --failed --no-pager
journalctl -p warning..alert -b --since '30 minutes ago' --no-pager
df -hT
free -h
ss -tulpn
ip -br addr
ip route
resolvectl status
top -b -n 1 | head -40

Decision sequence

  1. Is the host reachable and is time correct?
  2. Are critical services active and stable?
  3. Is there a recent kernel, package, deploy, or config change?
  4. Is a resource saturated: CPU, memory, disk, inode, I/O, network, DNS?
  5. Can you restore service by rollback, restart, scaling, freeing space, or failing over?
  6. What evidence should be preserved before cleanup?
