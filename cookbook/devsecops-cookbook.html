DevSecOps Cookbook

A practical field manual for designing, deploying, operating, and debugging a modern DevSecOps pipeline. The stack centers on Kubernetes, GitLab Runner, ArgoCD, Vault, Trivy, Cloudflare DNS, Prometheus, Grafana, Loki, and the log agents that connect them.

Operational Flow

A secure delivery path starts with a signed source event, produces a verifiable artifact, reconciles desired state, runs on hardened nodes, emits telemetry, and keeps credentials out of application images.

Flow
1 Commit triggers CI

GitLab Runner picks a job, checks out source, restores cache, builds, tests, and publishes artifacts.

2 Scan the artifact

Trivy scans image layers, packages, IaC, config, SBOMs, and secrets before promotion.

3 GitOps records intent

A deployment repo change updates Helm values, Kustomize overlays, or raw manifests.

4 ArgoCD reconciles

The application-controller compares Git state with live cluster state and applies drift corrections.

5 Kubernetes runs it

The API server, scheduler, controllers, kubelet, CNI, CSI, and container runtime execute the workload.

6 Observe and secure

Vault issues secrets, DNS publishes endpoints, and Prometheus, Grafana, and Loki expose runtime evidence.

Safety boundary: Treat DNS writes, Vault root tokens, cluster-admin kubeconfigs, forced ArgoCD syncs, and Kubernetes delete commands as production-impacting operations. Prefer short-lived credentials, scoped tokens, dry-runs, peer review, and explicit namespace targeting.

Practice 1: Container Orchestration & Node Management

Kubernetes is the runtime control system. `kubectl`, Helm, and kubelet-facing diagnostics are the operator interface for desired state, release packaging, and node inspection.

P1

Kubernetes

Declarative control plane for container workloads, networking, service discovery, storage attachment, and cluster policy.

control planedata planeruntime

Control plane

  • `kube-apiserver` is the HTTPS front door. It validates requests, applies admission, persists objects to `etcd`, and serves watch streams.
  • `etcd` stores desired and observed cluster state. Backups and quorum health are part of the security boundary.
  • `kube-scheduler` binds unscheduled Pods to Nodes based on resources, affinity, taints, topology, and policy.
  • `kube-controller-manager` runs reconciliation loops for Deployments, ReplicaSets, Nodes, Jobs, endpoints, and service accounts.

Data plane

  • `kubelet` watches PodSpecs assigned to its node and asks the container runtime through CRI to run containers.
  • `kube-proxy` or an eBPF replacement implements Service virtual IP routing.
  • CNI plugins attach Pod networking; CSI plugins attach persistent storage.
  • CoreDNS provides in-cluster DNS for Services and Pods.

Production baseline

  • Use managed Kubernetes or kubeadm with HA control plane nodes, externalized backups, and private API endpoints where possible.
  • Separate platform namespaces from application namespaces. Apply ResourceQuota, LimitRange, NetworkPolicy, and Pod Security Admission labels.
  • Keep `etcd` encrypted at rest, protect kubeconfig distribution, and rotate client certificates and service account tokens.
  • Install platform add-ons with Helm or GitOps, not ad-hoc mutable shell sessions.

Release hygiene

  • Pin Kubernetes minor versions in runbooks and validate add-on compatibility before upgrades.
  • Drain nodes before kernel/runtime upgrades and respect PodDisruptionBudgets.
  • Test admission policies and CRDs in a staging cluster before promoting to production.

kubectl config get-contexts
kubectl config use-context <cluster>
kubectl auth can-i create deployments --namespace <namespace>
kubectl create namespace <namespace> --dry-run=client -o yaml
kubectl label namespace <namespace> pod-security.kubernetes.io/enforce=restricted --overwrite

Where to look

  • Control plane: API server, scheduler, controller-manager, cloud-controller-manager, and `etcd` logs.
  • Node: `journalctl -u kubelet`, container runtime logs, CNI logs, CSI plugin Pods, and node conditions.
  • Workload: Pod events, readiness/liveness failures, image pull errors, admission denials, and Service endpoint mismatches.

Failure pattern

Start from desired state, then walk toward runtime: object exists, controller created child resources, scheduler bound the Pod, kubelet pulled and started containers, readiness passed, endpoints updated, and DNS/ingress routed traffic.

kubectl

The Kubernetes API client used for inspection, reconciliation, debug sessions, rollout control, and emergency operations.

clientREST APIdebug

`kubectl` reads kubeconfig, selects cluster/user/context, builds authenticated HTTPS requests, and talks to `kube-apiserver`. Output printers, JSONPath, server-side apply, watch streams, and exec/log subresources are client behaviors layered on top of Kubernetes API resources.

  • Install a kubectl version within one minor version of the cluster control plane.
  • Store kubeconfigs with least-privilege users, avoid long-lived cluster-admin files, and prefer OIDC or exec credential plugins.
  • Use explicit `--context` and `--namespace` in scripts to prevent accidental cross-cluster writes.
shell
kubectl version --client
kubectl config view --minify
kubectl config set-context --current --namespace=<namespace>
kubectl auth can-i '*' '*' --all-namespaces
shell
kubectl get pods -n <namespace> -o custom-columns=NAME:.metadata.name,NODE:.spec.nodeName,PHASE:.status.phase
kubectl get secret <secret> -n <namespace> -o jsonpath='{.type}'
kubectl patch deployment <deployment> -n <namespace> --type merge -p '{"spec":{"replicas":3}}'
kubectl exec -n <namespace> deploy/<deployment> -- env
kubectl port-forward -n <namespace> svc/<service> 8080:80

When kubectl fails, separate client, auth, network, and API problems. Use `kubectl --v=8` for HTTP trace detail, verify kubeconfig context, and test RBAC with `kubectl auth can-i` before assuming the workload is broken.

shell
kubectl --v=8 get ns
kubectl cluster-info
kubectl auth can-i get pods -n <namespace> --as <user>
kubectl get --raw='/readyz?verbose'

Helm

Kubernetes package manager for repeatable releases, chart templating, rollback metadata, and values-driven deployment.

chartsrelease statevalues

A Helm chart is a versioned bundle of templates, default values, metadata, and optional hooks/tests. Helm renders templates locally, submits Kubernetes manifests to the API server, and stores release history in cluster Secrets or ConfigMaps, enabling upgrades and rollbacks.

  • Pin chart versions and values files in Git; do not rely on floating chart latest in production.
  • Render with `helm template` and inspect diffs before cluster writes.
  • Keep environment-specific values small and explicit; use sealed or external secrets rather than plaintext secret values.
shell
helm repo add <repo> <chart_repo_url>
helm repo update
helm template <release> <repo>/<chart> -n <namespace> -f values.yaml
helm upgrade --install <release> <repo>/<chart> -n <namespace> --create-namespace -f values.yaml --atomic --timeout 10m
shell
helm list -A
helm status <release> -n <namespace>
helm get values <release> -n <namespace> --all
helm get manifest <release> -n <namespace>
helm history <release> -n <namespace>
helm rollback <release> <revision> -n <namespace>
helm uninstall <release> -n <namespace> --dry-run

Helm failures are usually invalid templates, missing CRDs, rejected admission, hook Jobs, or unhealthy workloads after apply. Inspect rendered manifests, release Secrets, hook Pods, and Kubernetes events.

shell
helm upgrade --install <release> <chart> -n <namespace> -f values.yaml --dry-run --debug
kubectl get secret -n <namespace> -l owner=helm,name=<release>
kubectl get jobs,pods -n <namespace> -l helm.sh/hook
kubectl get events -n <namespace> --sort-by=.lastTimestamp

kubeletctl

Security assessment client for kubelet APIs. Use it to verify node exposure, auth posture, and kubelet API behavior from controlled networks.

node APIassessment10250

kubeletctl talks directly to kubelet endpoints, commonly on HTTPS port `10250`. Those endpoints expose node health, Pod lists, metrics, logs, and in dangerous configurations exec-like access. In a hardened cluster, kubelet authentication, authorization, TLS, firewalling, and API server mediated access prevent direct abuse.

Controlled use only: Run kubeletctl from approved assessment hosts. Do not scan networks you do not own. Treat service account tokens and kubelet client certificates as sensitive credentials.

shell
kubeletctl version
kubeletctl --help
kubeletctl scan --cidr <node_cidr>
kubeletctl --server <node_ip> healthz
shell
kubeletctl --server <node_ip> pods
kubeletctl --server <node_ip> metrics
kubeletctl --server <node_ip> log
kubeletctl --server <node_ip> runningpods
kubeletctl --server <node_ip> --namespace <namespace> --pod <pod> --container <container> containerLogs
  • Connection refused usually means firewall, host networking, or kubelet port configuration blocks direct access.
  • HTTP 401/403 is the expected result on hardened nodes unless the client presents authorized credentials.
  • If unauthenticated endpoints respond with Pod data or exec capability, immediately restrict kubelet access and rotate exposed tokens.

Practice 2: GitOps & Continuous Delivery

Git is the delivery API. ArgoCD continuously compares desired state from Git with live state in Kubernetes and reconciles drift.

P2

ArgoCD

Declarative continuous delivery controller for Kubernetes applications sourced from Git, Helm charts, Kustomize, Jsonnet, or plain manifests.

GitOpsreconcilerollback

Core components

  • `argocd-server` serves API/UI and authenticates users through local accounts, SSO, or Dex.
  • `repo-server` clones Git and renders Helm, Kustomize, Jsonnet, or directory manifests.
  • `application-controller` watches Applications, compares desired and live state, computes health, and applies sync operations.
  • Redis caches application state and repository metadata.

Security model

  • AppProjects scope allowed repos, destinations, namespaces, and resource kinds.
  • Cluster credentials are stored as Kubernetes Secrets in the ArgoCD namespace.
  • Sync waves and hooks control ordering, but failed hooks can block delivery.
  • Install ArgoCD with the official manifests or Helm chart into a dedicated `argocd` namespace.
  • Use AppProjects to isolate teams and environments. Disable broad cluster-scoped writes unless required.
  • Prefer automated sync with pruning only after review gates, health checks, and rollback behavior are understood.
shell
kubectl create namespace argocd --dry-run=client -o yaml | kubectl apply -f -
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
kubectl rollout status deploy/argocd-server -n argocd
kubectl port-forward svc/argocd-server -n argocd 8080:443
shell
argocd login <argocd_host>
argocd app list
argocd app get <app>
argocd app diff <app>
argocd app sync <app> --prune
argocd app wait <app> --health --sync --timeout 600
argocd app history <app>
argocd app rollback <app> <revision_id>
yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: <app>
  namespace: argocd
spec:
  project: <project>
  source:
    repoURL: <git_repo_url>
    targetRevision: main
    path: environments/prod
  destination:
    server: https://kubernetes.default.svc
    namespace: <namespace>
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
  • Compare failures: repo render error, invalid manifests, missing CRDs, rejected admission, or destination RBAC denial.
  • Health failures: workload applied but ArgoCD health check reports Progressing, Degraded, or Missing.
  • Sync stuck: inspect hooks, sync waves, finalizers, and resources requiring deletion approval.
shell
kubectl logs -n argocd deploy/argocd-application-controller --since=30m
kubectl logs -n argocd deploy/argocd-repo-server --since=30m
kubectl logs -n argocd deploy/argocd-server --since=30m
argocd app get <app> --hard-refresh
argocd app resources <app>
argocd app manifests <app>

GitOps Flow

A delivery operating model where reviewed Git commits are the durable source of truth for runtime configuration.

promotiondriftreview

GitOps separates artifact creation from environment promotion. CI builds and signs images; a deployment repository records image tags, digests, Helm values, or Kustomize overlays; ArgoCD reconciles that intent. This creates an audit trail and makes rollback a Git or ArgoCD history operation.

shell
git checkout -b promote/<app>-<image_tag>
yq -i '.image.tag = "<image_tag>"' environments/prod/<app>/values.yaml
yq -i '.image.digest = "<image_digest>"' environments/prod/<app>/values.yaml
git diff
git commit -am "Promote <app> to <image_tag>"
git push origin promote/<app>-<image_tag>

If Git and live state disagree, decide whether the cluster is wrong or Git is stale. Never patch live state as the permanent fix; patching live state without committing the desired state creates repeating drift.

Practice 3: Continuous Integration & Build Automation

CI creates the evidence trail: source revision, tests, artifact digest, scan output, provenance, and deployable configuration updates.

P3

GitLab CI Runner

Agent that polls GitLab for jobs, executes them through an executor, uploads artifacts, and reports job state back to the coordinator.

CIexecutorsartifacts

Control path

  • The runner service authenticates to GitLab, asks for jobs, streams logs, and reports job status.
  • Registration and authentication tokens must be scoped and rotated like production credentials.
  • Runner tags, protected branches, and locked runners determine which jobs can execute on which infrastructure.

Execution path

  • Executors include Docker, Kubernetes, shell, SSH, virtual machine, and custom drivers.
  • The helper image handles repository checkout, artifact upload/download, and cache transfer.
  • Cache accelerates repeated dependency downloads; artifacts preserve build outputs and scan reports.
  • Use the Kubernetes executor for cluster-native isolation, or Docker executor for dedicated build hosts.
  • Use separate runners for protected branches, privileged builds, untrusted merge requests, and release signing.
  • Limit privileged mode, mount only required volumes, and use short-lived registry credentials.
shell
helm repo add gitlab https://charts.gitlab.io
helm repo update
helm upgrade --install gitlab-runner gitlab/gitlab-runner \
  -n gitlab-runner --create-namespace \
  --set gitlabUrl=<gitlab_url> \
  --set runnerRegistrationToken=<registration_token>
shell
gitlab-runner --version
gitlab-runner list
gitlab-runner verify
gitlab-runner status
gitlab-runner run --working-directory /var/lib/gitlab-runner
gitlab-runner unregister --name <runner_name>
yaml
stages:
  - test
  - scan
  - package

variables:
  IMAGE: <registry>/<project>:<commit_sha>

trivy:image:
  stage: scan
  image: aquasec/trivy:latest
  script:
    - trivy image --exit-code 1 --severity HIGH,CRITICAL "$IMAGE"
  artifacts:
    when: always
    paths:
      - trivy-report.json
  • Stuck jobs usually mean tag mismatch, runner offline, protected branch restrictions, or exhausted concurrency.
  • Image pull failures usually mean registry auth, network egress, DNS, or missing image architecture.
  • Cache failures are often object storage credentials, cache key churn, or unmounted cache volumes.
shell
kubectl logs -n gitlab-runner deploy/gitlab-runner --since=30m
kubectl describe pod -n gitlab-runner -l app=gitlab-runner
journalctl -u gitlab-runner --since "30 minutes ago"
gitlab-runner verify --delete

Practice 4: Secrets Management & Zero Trust

Secrets and vulnerability evidence should be short-lived, auditable, and policy-controlled. Vault issues secrets; Trivy validates what you ship and run.

P4

HashiCorp Vault

Central control plane for static secrets, dynamic credentials, encryption services, workload identity, leases, and audit trails.

secretsidentityleases

Core internals

  • Vault stores encrypted data behind a barrier and must be unsealed or auto-unsealed before serving requests.
  • Auth methods turn identities into tokens; policies attach capabilities to paths.
  • Secrets engines manage static KV, PKI, transit encryption, databases, cloud credentials, and dynamic leases.
  • Audit devices record request metadata and are mandatory for production traceability.

Kubernetes integration

  • Kubernetes auth validates service account JWTs against the Kubernetes API.
  • Vault Agent Injector, CSI provider, and Vault Secrets Operator are different ways to deliver secrets to Pods.
  • Use namespaces, policies, auth roles, and token TTLs to isolate applications.
  • Deploy with the official Helm chart or an external managed Vault; use HA mode for production.
  • Use integrated storage or supported HA storage, TLS everywhere, auto-unseal backed by a cloud KMS or HSM, and persistent audit logs.
  • Never put Vault root tokens in CI variables, Kubernetes Secrets, shell history, or Git.
shell
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update
helm upgrade --install vault hashicorp/vault -n vault --create-namespace -f vault-values.yaml
kubectl get pods -n vault
kubectl exec -n vault vault-0 -- vault status
shell
export VAULT_ADDR=<vault_addr>
vault status
vault login -method=oidc
vault secrets enable -path=kv kv-v2
vault kv put kv/<app>/config username=<username> password=<password>
vault kv get kv/<app>/config
vault token lookup
vault lease renew <lease_id>
vault audit enable file file_path=/vault/audit/audit.log
shell
vault auth enable kubernetes
vault write auth/kubernetes/config \
  kubernetes_host=<kubernetes_api_url> \
  kubernetes_ca_cert=@<ca_file>
vault policy write <app>-read ./policy.hcl
vault write auth/kubernetes/role/<app> \
  bound_service_account_names=<service_account> \
  bound_service_account_namespaces=<namespace> \
  policies=<app>-read \
  ttl=1h
  • `sealed` means Vault cannot decrypt storage; verify auto-unseal credentials and KMS access.
  • `permission denied` means token policy lacks capabilities for the requested path or auth role binding is wrong.
  • Kubernetes auth failures usually come from service account audience, issuer, namespace/name binding, or API connectivity.
shell
kubectl logs -n vault statefulset/vault --since=30m
kubectl describe pod -n vault vault-0
vault read sys/health
vault auth list
vault token capabilities <token> kv/data/<app>/config
vault audit list

Trivy

Security scanner for container images, filesystems, repositories, IaC misconfigurations, SBOMs, licenses, and secrets.

scanSBOMCI gate

Trivy downloads vulnerability and policy databases, inspects target artifacts, detects OS and language packages, evaluates misconfiguration rules, and reports findings with severity and package metadata. In CI, it becomes an admission gate before promotion.

  • Cache Trivy DB between CI jobs to reduce network dependency and speed up scans.
  • Use digest-pinned images for release gates and export SARIF or JSON for security evidence.
  • Fail on HIGH/CRITICAL for protected branches, but document exception handling for accepted risk.
shell
trivy --version
trivy image --download-db-only
trivy image --severity HIGH,CRITICAL --exit-code 1 <image>
trivy fs --scanners vuln,secret,misconfig .
trivy config ./manifests
trivy image --format cyclonedx --output sbom.cdx.json <image>
shell
trivy image --ignore-unfixed --severity HIGH,CRITICAL <image>
trivy image --format json --output trivy-image.json <image>
trivy repo --scanners vuln,secret,misconfig <git_repo_url>
trivy k8s --report summary cluster
trivy sbom --format spdx-json --output sbom.spdx.json <image>
trivy clean --scan-cache

Trivy failures are usually DB download issues, registry auth, rate limits, unsupported image architecture, or scan policy thresholds. Add `--debug`, validate registry login, and separate scanner errors from vulnerability findings.

shell
trivy image --debug <image>
docker login <registry>
trivy image --skip-db-update <image>
trivy image --timeout 10m <image>

Practice 5: Edge Networking & DNS

Cloudflare DNS, ExternalDNS, and cert-manager connect cluster services to names, edge routing, and certificates.

P5

Cloudflare DNS API

HTTP API for managing zones, DNS records, proxy mode, TTLs, and automation tokens used by CI, ExternalDNS, and cert-manager.

DNSAPI tokenedge

Cloudflare stores DNS records under zones. API tokens authorize actions like `Zone:DNS:Read` and `Zone:DNS:Edit`. Records can be DNS-only or proxied through Cloudflare edge. Automation should scope tokens to exact zones and avoid global keys.

  • Create one token per automation boundary: ExternalDNS, cert-manager DNS01, and manual break-glass scripts.
  • Use DNS-only records for validation and private origins unless Cloudflare proxy behavior is intended.
  • Store tokens in Vault or a sealed secret workflow, not plaintext manifests.
shell
curl -sS -X GET "https://api.cloudflare.com/client/v4/zones" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json"
shell
curl -sS -X GET "https://api.cloudflare.com/client/v4/zones/<cloudflare_zone_id>/dns_records?type=A&name=<fqdn>" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json"

curl -sS -X POST "https://api.cloudflare.com/client/v4/zones/<cloudflare_zone_id>/dns_records" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  --data '{"type":"A","name":"<fqdn>","content":"<ip_address>","ttl":300,"proxied":false}'
  • HTTP 403 usually means token scope, zone mismatch, or missing DNS edit permission.
  • DNS propagation problems can be stale recursive resolver cache, wrong TTL expectations, split-horizon DNS, or proxied vs DNS-only confusion.
  • For ACME DNS01, verify TXT record name and token scope before blaming cert-manager.

ExternalDNS

Kubernetes controller that watches Services and Ingresses and reconciles DNS records in providers such as Cloudflare.

controllerDNS syncCloudflare

ExternalDNS watches Kubernetes resources, builds a desired DNS record set, compares it against provider records, and applies changes. A TXT registry marks ownership so multiple clusters or controllers do not overwrite each other accidentally.

yaml
provider: cloudflare
policy: sync
domainFilters:
  - <domain>
env:
  - name: CF_API_TOKEN
    valueFrom:
      secretKeyRef:
        name: cloudflare-api-token
        key: api-token
txtOwnerId: <cluster>
shell
kubectl annotate ingress <ingress> -n <namespace> external-dns.alpha.kubernetes.io/hostname=<fqdn>
kubectl logs -n external-dns deploy/external-dns --since=30m
kubectl describe deploy -n external-dns external-dns
kubectl get ingress,svc -A -o wide

Check source resources, domain filters, provider credentials, TXT ownership records, and whether policy is `sync` or `upsert-only`. A typo in hostname annotation can create a valid record in the wrong place.

cert-manager

Kubernetes certificate controller that automates issuance, renewal, and ACME DNS01 validation with Cloudflare.

TLSACMEDNS01

cert-manager watches Certificate resources, uses Issuers or ClusterIssuers, creates ACME Orders and Challenges, writes DNS01 TXT records through a provider token, waits for validation, and stores issued key material in Kubernetes Secrets.

yaml
apiVersion: cert-manager.io/v1
kind: ClusterIssuer
metadata:
  name: letsencrypt-cloudflare
spec:
  acme:
    email: <email>
    server: https://acme-v02.api.letsencrypt.org/directory
    privateKeySecretRef:
      name: letsencrypt-cloudflare-account-key
    solvers:
      - dns01:
          cloudflare:
            apiTokenSecretRef:
              name: cloudflare-api-token
              key: api-token
shell
kubectl get clusterissuer,issuer -A
kubectl get certificate,certificaterequest,order,challenge -A
kubectl describe certificate <certificate> -n <namespace>
kubectl describe challenge <challenge> -n <namespace>
kubectl logs -n cert-manager deploy/cert-manager --since=30m

Failed issuance usually comes from ACME account problems, DNS01 TXT record creation failures, wrong hosted zone, Cloudflare token permissions, DNS propagation delay, or rate limits. Always inspect Certificate, Order, and Challenge resources together.

Practice 6: Observability (Monitoring & Logging)

Metrics, dashboards, logs, and alerting turn a deployment into an operable service. Prometheus answers what changed; Loki answers what happened; Grafana connects both for humans.

P6

Prometheus

Pull-based metrics and alerting toolkit with PromQL, local TSDB, service discovery, exporters, recording rules, and Alertmanager integration.

metricsPromQLalerts

Prometheus discovers scrape targets, pulls metrics over HTTP, stores samples in a local TSDB, evaluates PromQL rules, and sends alert state to Alertmanager. It is intentionally autonomous so it remains useful during outages.

  • Use kube-prometheus-stack for Kubernetes because it installs Prometheus Operator, Alertmanager, Grafana dashboards, ServiceMonitors, and common rules.
  • Keep label cardinality low; high-cardinality labels can exhaust memory and storage.
  • Set retention by time and size, plan remote write if long-term retention is required, and test alert routes.
shell
helm repo add prometheus-community https://prometheus-community.github.io/helm-charts
helm repo update
helm upgrade --install kube-prometheus-stack prometheus-community/kube-prometheus-stack \
  -n monitoring --create-namespace -f monitoring-values.yaml
shell
kubectl get prometheus,alertmanager,servicemonitor,prometheusrule -n monitoring
kubectl port-forward -n monitoring svc/kube-prometheus-stack-prometheus 9090:9090
promtool check config prometheus.yml
promtool check rules rules.yml
curl -sS "http://localhost:9090/api/v1/query?query=up"
promql
sum(rate(container_cpu_usage_seconds_total{namespace="<namespace>"}[5m])) by (pod)
sum(kube_pod_container_status_restarts_total{namespace="<namespace>"}) by (pod)
histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service))
  • Targets down: inspect ServiceMonitor selectors, scrape endpoint path/port, TLS/auth config, and NetworkPolicy.
  • Missing metrics: check exporter deployment, Service labels, and Prometheus target discovery page.
  • Memory growth: inspect high-cardinality labels and long query ranges.
shell
kubectl logs -n monitoring statefulset/prometheus-kube-prometheus-stack-prometheus --since=30m
kubectl describe servicemonitor -n monitoring <servicemonitor>
kubectl get endpoints -n <namespace> <service>
curl -sS "http://localhost:9090/api/v1/targets" | jq '.data.activeTargets[] | select(.health!="up")'

Grafana

Dashboard, alerting, and exploration layer for Prometheus metrics, Loki logs, traces, and operational runbooks.

dashboardsdatasourcesalerts

Grafana stores dashboards, users, folders, alert rules, datasources, and plugin metadata in its database. Query execution usually goes through datasource plugins to Prometheus, Loki, Tempo, or other backends; Grafana does not replace the telemetry stores.

  • Provision datasources and dashboards from ConfigMaps or chart values to keep changes reviewable.
  • Use SSO and team mappings; avoid shared admin accounts.
  • Back up Grafana database or keep dashboards fully provisioned from Git.
shell
kubectl get secret -n monitoring kube-prometheus-stack-grafana -o jsonpath='{.data.admin-password}' | base64 -d
kubectl port-forward -n monitoring svc/kube-prometheus-stack-grafana 3000:80
kubectl logs -n monitoring deploy/kube-prometheus-stack-grafana --since=30m

Dashboard failures are usually datasource URL/auth, folder permission, missing plugin, query timeout, or broken variable. Alert failures need separate checks for rule evaluation, contact point routing, and notification delivery.

Loki

Log aggregation system that indexes labels, stores compressed log chunks, and queries logs with LogQL.

logsLogQLlabels

Loki receives log streams, indexes labels instead of full text, writes compressed chunks to storage, and serves queries through queriers or query frontends. Distributed mode splits write, read, backend, and gateway concerns for scale.

  • Use Helm for monolithic, simple scalable, or microservices deployment mode based on retention and query volume.
  • Keep labels low-cardinality. Use structured metadata or parsed fields for high-cardinality values.
  • Plan object storage, retention, compactor, cache, and tenant strategy before production ingest.
shell
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update
helm upgrade --install loki grafana/loki -n monitoring -f loki-values.yaml
kubectl get pods -n monitoring -l app.kubernetes.io/name=loki
shell
logcli --addr=<loki_url> labels
logcli --addr=<loki_url> series '{namespace="<namespace>"}' --since=1h
logcli --addr=<loki_url> query '{namespace="<namespace>", app="<app>"} |= "error"' --since=30m
kubectl port-forward -n monitoring svc/loki 3100:3100
logql
sum by (pod) (rate({namespace="<namespace>"} |= "error" [5m]))
{namespace="<namespace>", app="<app>"} | json | level="error"
topk(10, sum by (pod) (count_over_time({namespace="<namespace>"}[15m])))
  • Ingest failures: check distributor logs, client push errors, rate limits, tenant headers, and label validation.
  • Query failures: check querier/query-frontend logs, object storage access, cache health, and excessive time ranges.
  • Storage failures: inspect compactor, index gateway, retention config, and object storage permissions.

Promtail / Grafana Alloy

Log collection agent layer. Promtail is now legacy; Alloy is the supported path for new Grafana log collection pipelines.

agentKubernetes logsmigration

Promtail runs as a DaemonSet, discovers Pods on each node, tails container log files, tracks positions, applies pipeline stages, attaches labels, and pushes streams to Loki. As of March 2, 2026, Promtail is EOL; keep this knowledge for existing estates and use Grafana Alloy for new deployments.

  • For new systems, deploy Grafana Alloy or the current Grafana Kubernetes monitoring chart.
  • For existing Promtail, keep pipeline stages small, avoid high-cardinality labels, and monitor dropped entries.
  • Mount `/var/log/pods` and `/var/lib/docker/containers` only as required by the runtime.
shell
helm repo add grafana https://grafana.github.io/helm-charts
helm repo update
helm upgrade --install alloy grafana/alloy -n monitoring -f alloy-values.yaml
kubectl get pods -n monitoring -l app.kubernetes.io/name=alloy
shell
kubectl logs -n monitoring daemonset/promtail --since=30m
kubectl logs -n monitoring daemonset/alloy --since=30m
kubectl describe pod -n monitoring -l app.kubernetes.io/name=alloy
kubectl get configmap -n monitoring -l app.kubernetes.io/name=alloy -o yaml

Agent problems are usually missing host log mounts, RBAC denial for Pod metadata, bad relabel rules, Loki push errors, tenant header mismatch, or label cardinality. Compare agent logs with Loki distributor logs to find whether data left the node.
