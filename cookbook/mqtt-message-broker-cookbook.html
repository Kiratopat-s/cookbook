MQTT & Message Broker Cookbook

A single-file operational cookbook for MQTT 3.1.1, MQTT 5.0, Mosquitto, EMQX, HiveMQ, secure broker access, CLI testing, Kubernetes orchestration, Prometheus telemetry, and the failure modes that show up at 03:17 when devices start flapping.

Atomic layerPackets, sessions, topic matching, inflight queues, retained state, wills, keep-alive, and WebSocket transport.
Broker layerMosquitto single-node patterns, EMQX Erlang clustering, HiveMQ Java platform operations, and broker bridges.
DevSecOps layerTLS, mTLS, ACLs, OIDC/JWT, Docker Compose, Kubernetes operators, metrics, logs, and runbooks.

When to use this

  • You need a broker topology for IoT telemetry, command/control, fleet state, or factory event streams.
  • You are debugging QoS, duplicate delivery, retained-state surprises, LWT status, or keep-alive disconnects.
  • You are choosing between lightweight Mosquitto and clustered enterprise brokers such as EMQX or HiveMQ.

Prerequisites

  • Shell access with Docker, Docker Compose, kubectl, and optionally Helm.
  • MQTT clients: mosquitto_pub, mosquitto_sub, and mqtt from MQTT CLI.
  • A private CA, DNS names, and separate staging/prod broker endpoints for security testing.
Safety

Never test destructive retained-message cleanup, ACL changes, bridge loops, or cluster leave commands against production without a named rollback. MQTT state is small, but a bad wildcard or retained delete can change the behavior of every device that reconnects.

Practice 1

Protocol Architecture & Message Lifecycle

MQTT is a client-server publish/subscribe protocol over an ordered, lossless, bidirectional transport. The broker is not just a relay: it owns topic matching, retained records, session state, inflight packet identifiers, offline queues, and abnormal-disconnect signaling.

Atomic ground base architecture

  1. Transport accept. A TCP, TLS, or WebSocket listener accepts the socket. The broker reads the first MQTT control packet from the stream and decodes fixed header, remaining length, variable header, properties, and payload.
  2. CONNECT handshake. The broker validates protocol version, client ID, clean session or clean start flags, keep-alive, credentials, optional will, and MQTT 5 properties such as session expiry and receive maximum.
  3. Session attach. The broker either creates a fresh session or resumes an existing one. If another socket already owns the same client ID, the older connection is usually taken over and disconnected.
  4. Publish ingress. A PUBLISH packet is authorized, decoded, assigned a packet identifier for QoS 1 or 2, optionally stored as retained state, then routed through the topic tree.
  5. Topic match. The broker compares the publish topic against subscription filters, including + single-level and # multi-level wildcards. Shared subscriptions add a load-balancing group after the match.
  6. Delivery egress. Each subscriber gets a packet at the negotiated QoS, capped by receive maximum, queue limits, ACL checks, and session expiry. QoS 1 and 2 messages live in inflight state until acknowledged.
publisher socket
 CONNECT(client_id=device-17, clean_start=false, keep_alive=60)
 PUBLISH(topic=plant/line1/temp, qos=1, packet_id=42, payload=21.8)

 broker core

 subscriber socket

QoS levels and inflight state

QoSContractPacket exchangeOperational costUse it for
0At most once. No retry, no broker acknowledgement.PUBLISHLowest CPU, memory, wire chatter.High-rate sensor readings where the next sample supersedes the last.
1At least once. Delivery is retried until acknowledged, so duplicates are possible.PUBLISH -> PUBACKPacket identifier plus retransmit timer on each sender.Commands, state changes, alerts, and most production telemetry.
2Exactly once between a client and the broker session.PUBLISH -> PUBREC -> PUBREL -> PUBCOMPHighest latency and session state on both sides.Billing-like operations where duplicate processing is worse than latency.
Inflight mechanics and duplicate behavior

Inflight state is indexed by packet identifier per client session, not globally. For QoS 1, a sender keeps the PUBLISH until PUBACK arrives. For QoS 2, both sides hold state across the four-step handshake. If a socket drops before completion and the session persists, retransmission resumes on reconnect with the DUP flag set.

client outbound inflight[77] = PUBLISH(payload_hash=9b2a, state=await_pubrec)

 receive PUBREC id=77
 state = await_pubcomp

 receive PUBCOMP id=77
 delete inflight[77]

 broker inbound inflight[77] = received_not_released

 send PUBREC id=77

 send PUBCOMP id=77
 delete inflight[77]
Design

QoS 2 gives exactly-once processing for the MQTT hop. It does not make your downstream database or webhook idempotent. Add an application message ID when an action must be globally deduplicated.

MQTT 3.1.1 vs MQTT 5.0 session behavior

MQTT 3.1.1 Clean Session clean_session=true means delete old session state and discard state on disconnect. false means keep subscriptions and queued QoS 1/2 messages for that client ID.
MQTT 5.0 Clean Start + Session Expiry clean_start=true controls the start of the connection. session_expiry_interval controls how long the broker keeps state after disconnect, including 0 for immediate expiry.

Session Present appears in CONNACK. It tells the client whether the broker resumed an existing session. A reconnecting durable subscriber should treat session_present=false as a signal to resubscribe because the broker no longer has its subscriptions.

Broker Internals & Single-Node Setups

Eclipse Mosquitto is the sharp, small-footprint broker to use for labs, gateways, edge nodes, single-host services, and bridge patterns. Its constraints are useful: explicit listeners, clear config files, simple persistence, and Unix-friendly operations.

Bridge architecture

A Mosquitto bridge is a broker-to-broker client connection. It subscribes and republishes across topic prefixes, usually from constrained edge networks into a regional broker or cloud broker. The failure mode to fear is a loop: the same message is bridged out, renamed, then bridged back as a new publish.

connection plant-east-to-core
address core-broker.example.net:8883
bridge_protocol_version mqttv50
try_private false
start_type automatic
cleansession false
local_clientid bridge-plant-east
remote_clientid bridge-plant-east
keepalive_interval 45
restart_timeout 10 120

remote_username bridge-plant-east
remote_password ${BRIDGE_PASSWORD}
bridge_cafile /mosquitto/config/certs/ca.pem
bridge_certfile /mosquitto/config/certs/plant-east.pem
bridge_keyfile /mosquitto/config/certs/plant-east-key.pem
bridge_insecure false

notifications true
notification_topic $SYS/broker/connection/plant-east-to-core/state

# Send local telemetry to the core broker.
topic telemetry/# out 1 tenants/plant-east/ tenants/plant-east/

# Pull cloud commands back to the edge broker.
topic commands/# in 1 tenants/plant-east/ tenants/plant-east/
Bridge loop

Do not bridge # both between two brokers unless you have deliberate topic rewriting that prevents loops. Prefer one directional telemetry path and one narrow command path.

Practice 3

Enterprise Clustering & High Availability

Clustered brokers add distributed routing tables, membership protocols, load balancers, rolling upgrades, operator reconciliation, and back-pressure controls. The hard part is not accepting a million sockets; it is keeping session state, routing metadata, and downstream integrations predictable during churn.

EMQX internals: Erlang, Mria, Ekka, and the rule engine

EMQX is built on Erlang/OTP. Each broker node is an Erlang node such as emqx@emqx1, authenticated with a shared cookie. In EMQX 5 and later, large clusters use core and replicant roles: core nodes own writes to cluster metadata, replicants serve client sessions and receive replicated routing metadata through the RLOG/Mria path.

MriaMetadata replicationProvides the replicated table layer used for routing metadata, durable state, and core-to-replicant replication patterns.
EkkaCluster membershipSupplies Erlang clustering and discovery plumbing historically used across EMQX node membership flows.
Rule engineSQL over broker eventsFilters MQTT events and messages, transforms fields, then executes actions such as republish, webhook, Kafka, PostgreSQL, S3, or other sinks.
client on emqx2 publishes tenants/a/devices/17/temp
emqx2 checks local ACL and receive maximum
emqx2 matches topic tree replicated from core nodes
routing table says subscribers live on emqx1 and emqx3
emqx2 forwards delivery work to emqx1 and emqx3
each target node checks its local subscription table
each target node writes to its own connected subscriber sockets

HiveMQ architecture and extension framework

HiveMQ is a Java-based enterprise broker. The operational mental model is a non-blocking network path backed by JVM worker pools, persistence, distributed cluster replication, and an extension framework. The "thread-per-core" lesson is that broker throughput comes from event loops and bounded worker pools sized around available cores, not from one thread per MQTT connection.

NetworkEvent-loop oriented I/OLong-lived sockets are multiplexed through broker I/O threads. Slow extension code must not block the hot path.
Extension SDKInterceptors and servicesExtensions authenticate, authorize, intercept publish/subscribe events, add metrics, and integrate systems.
ClusterShared broker stateCluster discovery, replication, and health checks keep sessions and routing available across broker nodes.

Practice 4

Production Security

MQTT security is layered: network segmentation, TLS, identity, authorization, message design, broker limits, and auditability. The best policy is boring: authenticate every client, authorize by tenant and client ID, deny broad wildcards, rotate credentials, and make abnormal disconnects visible.

TLS and mutual TLS

Use TLS on every untrusted network. Use mTLS for device identity when you can provision client certificates. Use username/password or JWT only over TLS. Never put long-lived broker passwords in process command lines in shared systems; prefer config files, secret mounts, or prompt-based CLI flows.

$ openssl req -x509 -new -nodes -key ca.key -sha256 -days 3650 \

OAuth2 and OIDC integration

MQTT has no built-in OAuth handshake. Brokers integrate OAuth2/OIDC by validating bearer tokens passed through username/password fields, enhanced authentication, custom extensions, or a broker-specific JWT/OIDC authenticator. Treat access tokens as credentials: short TTL, TLS only, narrow claims, no logs.

EMQX JWT/JWKS authentication 
authentication = [
  {
    mechanism = jwt
    backend = jwks
    endpoint = "https://idp.example.com/.well-known/jwks.json"
    refresh_interval = 300s
    from = password
    verify_claims = {
      aud = "mqtt-prod"
      iss = "https://idp.example.com/"
    }
    acl_claim_name = "mqtt_acl"
  }
]
$ export ACCESS_TOKEN="$(oidc-token-command print-token)"
$ mosquitto_pub -h broker.example.net -p 8883 --cafile ca.pem \
  -u device-17 -P "$ACCESS_TOKEN" \
  -t 'tenants/acme/devices/device-17/telemetry' \
  -q 1 -m '{"temp":21.8}'
HiveMQ extension-based token validation

Use the HiveMQ Enterprise Security Extension or a custom Extension SDK authenticator to validate JWTs against an IdP JWKS endpoint. Keep the authenticator asynchronous or cached so token validation cannot block broker I/O threads.

  publish tenants/acme/devices/device-17/telemetry

Credential handling

Do not paste real tokens, passwords, private keys, or client certificates into chat, issue trackers, or logs. When you need reproducible diagnostics, use a synthetic token signed by a staging key and a broker endpoint that cannot reach production devices.

Runbooks

Incident Runbooks

These are short paths for the ugly cases: auth storms, timeout storms, queue buildup, bridge loops, and missing retained state.

Auth failure storm

  1. Check whether failures are isolated to one tenant, one app release, one certificate CA, or every client behind a load balancer.
  2. Inspect broker logs for CONNACK reason codes, authentication_failed, authorization_permission_denied, or TLS verify failures.
  3. Run a known-good CLI connection from the same network path as the failing clients.
  4. If OIDC/JWKS is involved, confirm IdP health, JWKS rotation, clock skew, token audience, issuer, and expiry.
$ mosquitto_sub -h broker.example.net -p 8883 --cafile ca.pem \
  -u ops-probe -P "$MQTT_PASSWORD" \
  -t '$SYS/broker/uptime' -C 1 -W 5 -d

Keep-alive timeout storm

  1. Compare client keep-alive values with load balancer, firewall, NAT, and WebSocket proxy idle timeouts.
  2. Check broker CPU, garbage collection, Erlang scheduler utilization, JVM blocked threads, and disk I/O.
  3. Confirm reconnect jitter. Synchronized reconnect loops can look like broker failure while being a client rollout bug.
$ mqtt sub -h broker.example.net -p 8883 -s --cafile ca.pem \
  -i keepalive-probe -k 30 \
  -t '$SYS/broker/uptime' -q 0 -d

Bridge loop or message flood

  1. Temporarily disable bridge egress for the suspected connection, not the whole broker, if your config and change process allow it.
  2. Look for the same payload moving across rewritten topics repeatedly.
  3. Check retained state and rules that republish to a topic matched by their own input filter.
  4. Restore with asymmetric topic prefixes and explicit in/out bridge rules.
$ mosquitto_sub -h broker.example.net -t 'tenants/acme/#' -v -W 20 \
  | awk '{print $1}' \
  | sort \
  | uniq -c \
  | sort -nr \
  | head
