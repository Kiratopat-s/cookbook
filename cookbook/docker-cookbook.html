CH 01 · Foundations

What problem containers solve

Before any command: what a container is, what it is not, and why the industry moved from virtual machines to this.

The problem: “works on my machine”

Mental modelA container is a shipping container for software. You pack the app together with its runtime, libraries, and config into one sealed, standardized box. Any machine with a container runtime can run the box identically — the box doesn't care what ship (server) it's on, and the ship doesn't care what's inside the box.

Software breaks between machines because the app is only half the story — the other half is its environment: OS libraries, language runtime versions, environment variables, file paths, system packages. Traditionally that environment lived on the host and drifted over time. Containers fix this by making the environment part of the artifact you ship.

  • Image — the frozen, immutable package: your app + everything it needs, down to (but not including) the kernel.
  • Container — a running instance of an image, with its own isolated view of processes, network, and filesystem.
  • Registry — a server that stores and distributes images (Docker Hub, GHCR, ECR, a private one).
SENIOR LENS

Why did this design win? Not because isolation was new (VMs had it) but because containers made the artifact the unit of shipping. “Build once, run anywhere the kernel matches” collapsed the gap between dev, CI, and prod — the same bytes run in all three. When you evaluate any container decision later (base image, build strategy, orchestration), ask: does this keep the artifact immutable and reproducible? That's the property everything else depends on.

Containers vs virtual machines

A VM virtualizes hardware — each VM boots a full guest OS with its own kernel. A container virtualizes the operating system — all containers share the host's kernel and are just normal Linux processes wearing very good blindfolds.

Virtual machineContainer
IsolatesHardware (own kernel per VM)OS resources (shared kernel)
Boot timeSeconds–minutesMilliseconds (it's just a process)
SizeGBs (full OS disk)MBs (app + libs only)
DensityTens per hostHundreds per host
Isolation strengthStrong (hardware boundary)Good, kernel-enforced — but a kernel exploit escapes it
OS flexibilityAny guest OSSame kernel family as host (Linux containers need a Linux kernel)
INFO

So how does Docker run on macOS / Windows? It cheats honestly: Docker Desktop runs a tiny Linux VM (via Apple's Virtualization framework or WSL 2), and all your “local” containers actually run inside that hidden Linux VM. This explains many Mac/Windows quirks — file-sharing performance, host.docker.internal, and why --network host historically behaved differently there.

SENIOR LENS

When juniors pick wrong: containers are not “lightweight VMs” and shouldn't be treated as pet servers you SSH into and mutate. If your workflow is “exec in and edit files”, you've recreated the drift problem. Rebuild the image instead. Conversely, when you truly need a different kernel, strong multi-tenant isolation, or to run Windows-on-Linux — that's a VM's job (or both: VMs running containers, which is exactly what every cloud does).

CH 02 · Foundations

The atomic layer: what a container is made of

“Container” is not a kernel object. It's a recipe of three Linux features — namespaces (what you can see), cgroups (what you can use), and union filesystems (what you stand on) — plus a few security seasonings.

Namespaces — isolation of visibility

Mental modelA namespace is a pair of glasses the kernel puts on a process. Through PID-namespace glasses, your process sees itself as PID 1 and sees no other processes. Through network-namespace glasses, it sees its own private network stack. The host sees everything; the process sees only its slice.
NamespaceIsolatesEffect inside a container
pidProcess IDsYour app is PID 1; it cannot see or signal host processes.
netNetwork stackOwn interfaces, own IP, own routing table, own localhost, own port space.
mntMount pointsOwn filesystem tree — its / is the image's root, not the host's.
utsHostname / domainContainer gets its own hostname (default: the container ID).
ipcShared memory, semaphoresCan't peek into host or sibling IPC objects.
userUID / GID mappingsCan map “root inside” to an unprivileged host user. Off by default in Docker (opt-in via userns-remap / rootless mode).
cgroupCgroup tree viewContainer sees only its own resource-control subtree.
timeBoot/monotonic clocksRarely used; wall-clock time is not namespaced — containers share the host clock.

You can watch Docker do this. Compare the process view inside and outside:

# Inside a container, your process is PID 1 and it is alone:
$ docker run --rm alpine ps aux
PID   USER     COMMAND
  1   root     ps aux

# On the host, that same process is just a regular PID among thousands.
Deep dive — build a “container” by hand with unshareoptional

Namespaces are plain kernel syscalls (clone, unshare, setns). This is a container with no Docker involved:

# New PID + mount + UTS + net namespaces, running a shell:
$ sudo unshare --pid --fork --mount --uts --net --mount-proc sh

# Inside that shell:
$ hostname minibox        # only changes THIS namespace's hostname
$ ps aux                  # ...you are PID 1, nothing else visible
PID   USER     COMMAND
  1   root     sh
$ ip addr                 # only a lonely, down loopback interface

Everything Docker adds on top — the image filesystem via pivot_root, a veth network cable, cgroup limits — is orchestration around these primitives. This is why containers start instantly: nothing “boots”, the kernel just tags a process.

Cgroups — limits on consumption

Mental modelNamespaces control what you can see; cgroups control what you can spend. A control group is an accounting bucket the kernel attaches to processes: CPU time, memory bytes, disk I/O, process count — each metered and capped per bucket.
# Hard-cap memory at 256 MB and CPU at 1.5 cores:
$ docker run -d --name capped --memory=256m --cpus=1.5 nginx

# Watch live usage against those limits:
$ docker stats capped
NAME     CPU %   MEM USAGE / LIMIT   MEM %   NET I/O   PIDS
capped   0.02%   8.2MiB / 256MiB     3.2%    1kB/0B    9
WATCH OUT

Exceeding --memory is fatal, not throttled. The kernel OOM-killer terminates the process → container exits with code 137 (128 + SIGKILL 9). Exceeding --cpus merely throttles. If a container “randomly dies” under load, check docker inspect for "OOMKilled": true before blaming your code.

SENIOR LENS

Why always set limits in shared environments: without them, one leaking container can starve the host and take every neighbor down with it — the noisy-neighbor incident every on-call engineer eventually meets. Limits turn “the whole node died” into “one container restarted”. Kubernetes requests/limits are this exact mechanism with a scheduler on top.

Union filesystem (OverlayFS) — layered storage

Mental modelTransparent sheets on a projector. Each image layer is a read-only sheet; stack them and you see one combined picture. The container adds a single blank writable sheet on top — all your changes land there, the sheets below are never touched.

Three consequences that explain most Docker behavior:

  • Copy-on-write: modifying a file from a lower layer first copies it up to the writable layer. Cheap for configs; expensive for a 2 GB file you touch one byte of.
  • Deletes are masks: deleting a lower-layer file writes a whiteout marker on top. The bytes still exist in the layer — this is why secrets baked into an early layer are never removed by a later RUN rm.
  • Layers are shared: 50 containers from one image share the same read-only layers on disk and in page cache. That's the density win.
$ docker run -d --name web nginx
$ docker exec web sh -c 'echo hi > /new.txt && rm /etc/nginx/fastcgi_params'

# A = added, C = changed, D = deleted — the diff of the writable layer:
$ docker diff web
A /new.txt
C /etc/nginx
D /etc/nginx/fastcgi_params

The security seasonings: capabilities, seccomp, root

Isolation isn't only visibility — Docker also strips power from the container's processes:

  • Capabilities — root's power is split into ~40 slices (CAP_NET_ADMIN, CAP_SYS_TIME, …). Containers keep a small default set; “root in a container” can bind port 80 but cannot, say, change the host clock or load kernel modules.
  • Seccomp — a syscall firewall. Docker's default profile blocks ~40 dangerous syscalls (e.g. reboot, mount, ptrace variants).
  • AppArmor / SELinux — mandatory access-control policies on top, distro-dependent.
DANGER

--privileged disables essentially all of the above and hands the container the host. It exists for tools like Docker-in-Docker and device tinkering — never for ordinary apps. The same is true of mounting /var/run/docker.sock into a container: whoever holds the socket owns the host.

SENIOR LENS

The honest security summary: a container is a locked room in a shared house; a VM is a separate house. Containers are excellent at containing accidents and decent at containing attackers, but the shared kernel is a shared fate — one kernel exploit opens every room. That's why multi-tenant clouds run untrusted containers inside micro-VMs (Firecracker, gVisor). For your own workloads, the defaults + chapter 14's hardening are the right posture.

CH 03 · Foundations

Docker's architecture: who actually runs your container

docker the command never runs containers. It's a thin client talking to a chain of daemons, ending at a small binary that performs the kernel tricks from chapter 02 and exits.

The execution pipeline

Mental modelA restaurant. docker CLI is the waiter taking your order. dockerd is the front-of-house manager (API, images, networks, volumes). containerd is the kitchen manager supervising every dish. runc is the cook who actually makes one dish (creates one container) and immediately moves on. A small shim stays at each table so the kitchen manager can be restarted without dropping any dish.
ComponentJobWorth knowing
docker (CLI)Turns your command into REST callsCan target remote daemons: docker context / DOCKER_HOST=ssh://user@server. The CLI is fully replaceable.
dockerdThe Docker Engine API: builds, image storage, networks, volumes, logsListens on the Unix socket /var/run/docker.sock. Access to the socket = root on the host.
containerdSupervises container lifecycle: start, stop, pause, statusA CNCF project used far beyond Docker — it's the default runtime under Kubernetes too.
containerd-shimTiny per-container parent: holds stdio, reports exit codeLets dockerd/containerd restart or upgrade without killing your containers.
runcCreates one container from an OCI spec (namespaces, cgroups, pivot_root), then exitsThe reference OCI runtime. Swappable: gVisor's runsc, Kata (micro-VMs), etc.
 
$ docker version          # note the separate Client and Server sections
$ docker info             # daemon-wide view: runtime, storage driver, cgroup version

# The chain, live (Linux):
$ ps -e -o pid,ppid,cmd | grep -E 'dockerd|containerd|shim' | grep -v grep
  612     1  /usr/bin/dockerd -H fd://
  655     1  /usr/bin/containerd
 4102     1  /usr/bin/containerd-shim-runc-v2 -namespace moby -id 9f2c...
SENIOR LENS

Why layered like this? Separation of failure domains and standards. Because the pieces speak open specs (below), any layer is swappable — which is exactly what happened: Kubernetes dropped dockerd and talks to containerd directly ("Docker deprecation" panic of 2020 — your images were always fine, only the middleman changed). If you understand this diagram, that news cycle is a shrug.

Junior trap: assuming docker commands run "where I am". They run where the daemon is. With a remote DOCKER_HOST, docker build . still sends your local folder as context, but -v /path bind-mounts a path on the server.

OCI — the standards that make it all swappable

The Open Container Initiative defines three small specs. Memorize what each covers and vendor names stop mattering:

SpecStandardizesIn practice
Image specWhat an image is: layer tarballs + manifest + config JSONWhy an image built by Docker runs on Podman, containerd, Kubernetes…
Runtime specHow to run a filesystem bundle: config.json → namespaces, cgroups, mountsWhat runc (and drop-in replacements) implement.
Distribution specThe registry HTTP API: push/pull by digestWhy Docker Hub, GHCR, ECR, Harbor are interchangeable endpoints.

What actually happens on docker run

$ docker run -d -p 8080:80 nginx:1.27
  1. CLI → API. The CLI POSTs "create a container from nginx:1.27" to dockerd over the socket.
  2. Image resolution. No local copy? dockerd pulls from Docker Hub: fetch manifest, then each missing layer by digest, verifying checksums. Layers you already have are skipped — that's why second pulls are fast.
  3. Filesystem assembly. Storage driver (overlay2) mounts the layers read-only and adds a fresh writable layer → the container's /.
  4. Network plumbing. A network namespace is created; a veth cable connects it to the docker0 bridge; an IP is assigned; for -p 8080:80 the daemon programs NAT so host:8080 → container:80.
  5. Handoff. dockerd → containerd → shim → runc, which applies the OCI spec (namespaces, cgroups, mounts, capabilities), execves the image's configured command (nginx -g 'daemon off;')… and exits. The shim stays as the process's parent.
  6. You get a container ID. -d means "don't attach my terminal" — the container was never "in the background" or "foreground" itself; only your attachment differs.
TIP

Internalize step 6: docker run = docker create + docker start (+ attach unless -d). Every lifecycle command in chapter 07 is just poking a phase of this pipeline.

CH 04 · Foundations

Images: layers, tags, digests, registries

An image is not a disk file — it's a stack of read-only layer archives plus a small JSON config, all addressed by cryptographic hash.

Anatomy of an image

Mental modelImage : container :: class : instance. The image is the frozen blueprint (layers + config). Every docker run stamps out a new instance with its own writable layer, PID space, and IP. Delete the instance, the blueprint remains; run it 50 times, the blueprint is stored once.
  • Layers — gzipped tar archives of filesystem diffs, each identified by its SHA-256 digest.
  • Config JSON — default Cmd/Entrypoint, Env, exposed ports, working dir, and the ordered list of layer digests.
  • Manifest — the registry-side index tying config + layers together; multi-arch images have a manifest list (one entry per platform — how arm64 Macs and amd64 servers pull the same tag and get different bits).
# The layer recipe (bottom-up) with the Dockerfile step that made each:
$ docker history nginx:1.27

# The config JSON — defaults your containers inherit:
$ docker image inspect nginx:1.27 --format '{{json .Config}}' | head -c 400

# Layer digests:
$ docker image inspect nginx:1.27 --format '{{json .RootFS.Layers}}'

Image names, tags, and digests

registry.example.com:5000 / team / api : 1.4.2
└────────── registry ──────────┘ └repo┘ └tag┘

nginx            →  docker.io/library/nginx:latest   (all defaults filled in)
ghcr.io/org/app  →  GitHub Container Registry
app@sha256:9f8e… →  by digest: immutable, exact bytes forever
ReferenceNatureUse it for
app:1.4.2Mutable pointer. A tag is a sticky note; whoever pushes next moves it.Human-friendly day-to-day pulls.
app@sha256:…Immutable identity. Content-addressed — same digest ⇒ same bytes.Production pinning, supply-chain integrity, reproducible builds.
app:latestJust a tag named "latest" — no magic, no guarantee of newest.Local tinkering only. See warning below.
WATCH OUT

:latest lies twice. (1) It's only the default label applied when no tag is given — not "most recent". (2) docker run won't re-pull it if any local image has that tag, so machines silently run different code under the same name. In anything shared: pin versions, and for production consider pinning digests.

SENIOR LENS

Why content-addressing is the load-bearing idea: because layers are named by their hash, layers dedupe globally (a hundred images FROM the same base share one copy), pulls are integrity-verified by construction, and caches can never be "stale, but look fresh". When something feels magical about Docker's speed, the answer is usually "digests".

Registries

A registry is an HTTP server implementing the OCI distribution spec. docker pull = GET manifest, then GET each layer blob you don't have; docker push is the reverse. Common ones: Docker Hub (docker.io, the default), GHCR, AWS ECR, Google Artifact Registry, self-hosted Harbor or registry:2.

$ docker run -d -p 5000:5000 --name registry registry:2
$ docker tag myapp:1.0 localhost:5000/myapp:1.0
$ docker push localhost:5000/myapp:1.0
INFO

Docker Hub rate-limits anonymous pulls. In CI, authenticate (docker login) or mirror base images into your own registry — the classic cause of builds that fail only on busy afternoons with toomanyrequests.

CH 05 · Practice

Working with images — the CLI

Pull, list, tag, ship, inspect, and clean up images. Modern Docker groups these under docker image …; the short legacy aliases (docker pull, docker rmi…) are identical.

Getting images: pull & search

$ docker pull nginx:1.27              # explicit tag — do this
$ docker pull nginx                   # implies :latest
$ docker pull nginx@sha256:9f8e…      # exact bytes, forever
$ docker pull --platform linux/amd64 nginx:1.27   # force arch (e.g. on Apple Silicon)

$ docker search postgres              # quick Hub search (stars, official flag)
TIP

Prefer official images (docker.io/library/* — the ones with no user prefix, like postgres, redis, node) and Docker Verified Publisher images. Random personal images are the container equivalent of curl-piping a stranger's shell script.

Listing & identifying

$ docker images                        # alias: docker image ls
REPOSITORY   TAG      IMAGE ID       CREATED       SIZE
nginx        1.27     92b11f67ab2b   2 weeks ago   188MB
myapp        dev      1f3c9a7e0d21   3 hours ago   142MB

$ docker images -a                     # include intermediate build layers
$ docker images --digests              # show sha256 digests
$ docker images --filter dangling=true # untagged <none> leftovers
$ docker images --format '{{.Repository}}:{{.Tag}}  {{.Size}}'   # scriptable output

Dangling images (<none>:<none>) appear when a rebuild reuses a tag: the old image loses its name but keeps its bytes. They're the first thing to prune when disk fills up.

Tagging & pushing

docker tag creates a new name for the same image — zero bytes copied, just another sticky note on the same digest. Registries are part of the name, so “where it pushes” is decided by how you tag.

$ docker build -t myapp:1.4.2 .

# Same image, three names (note: tag ≙ alias, not copy):
$ docker tag myapp:1.4.2 ghcr.io/acme/myapp:1.4.2
$ docker tag myapp:1.4.2 ghcr.io/acme/myapp:latest

$ docker login ghcr.io                 # once per registry (token/password prompt)
$ docker push ghcr.io/acme/myapp:1.4.2
$ docker push ghcr.io/acme/myapp:latest
$ docker logout ghcr.io
WATCH OUT

denied: requested access to the resource is denied on push almost always means the image name lacks your registry/namespace prefix (you tried to push to docker.io/library/) or you aren't logged in to that registry. Fix the tag first, then the login.

Inspecting: history & metadata

$ docker history myapp:1.4.2           # layer-by-layer: which step added which MBs
$ docker history --no-trunc myapp:1.4.2

$ docker image inspect myapp:1.4.2 --format '{{.Config.Entrypoint}} {{.Config.Cmd}}'
$ docker image inspect myapp:1.4.2 --format '{{json .Config.Env}}'
$ docker image inspect myapp:1.4.2 --format '{{.Architecture}}/{{.Os}}'
SENIOR LENS

Make docker history a reflex. It answers “why is this image 1.2 GB?” in five seconds (you'll usually find a fat RUN layer or an accidental COPY . . of node_modules), and it reveals whether a third-party image does anything surprising. Reading history is also the fastest way to learn Dockerfile technique from well-built official images.

Removing images

$ docker rmi myapp:dev                 # alias: docker image rm — removes ONE tag;
                                       # bytes go only when the last tag is gone
$ docker rmi 1f3c9a7e0d21              # by ID (fails if containers still use it; -f to force)

$ docker image prune                   # delete dangling images
$ docker image prune -a                # delete ALL images not used by any container
$ docker image prune -a --filter "until=168h"   # …older than 7 days
DANGER

docker image prune -a removes every image without a running/stopped container attached — including ones you'll want tomorrow (they'll re-pull, but slowly, and private ones need auth). On laptops it's usually fine; on shared build hosts, prefer the --filter until=… form.

CH 06 · Practice

Building images — Dockerfile & BuildKit

A Dockerfile is a build script where almost every line mints a cached, shareable filesystem layer. Master three things — instruction semantics, cache ordering, multi-stage builds — and your images become small, fast, and reproducible.

Anatomy of a Dockerfile

Mental modelEach instruction runs in a temporary container, and its filesystem diff is snapshotted as a layer. The next instruction starts from that snapshot. The final snapshot chain + a config JSON = your image. Nothing “executes” at run time except ENTRYPOINT/CMD.
# syntax=docker/dockerfile:1

FROM python:3.12-slim               # base image = your starting layer stack

WORKDIR /app                        # cd for all following steps (creates the dir)

COPY requirements.txt .             # copy the dependency manifest FIRST (cache!)
RUN pip install --no-cache-dir -r requirements.txt

COPY . .                            # now the source code (changes most often → last)

ENV PYTHONUNBUFFERED=1              # runtime env var, baked into the image
EXPOSE 8000                         # documentation: “this app listens on 8000”

RUN useradd --create-home appuser   # never ship as root (ch 14)
USER appuser

CMD ["python", "-m", "uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
$ docker build -t myapi:1.0 .        # “.” = build context: the files sent to the builder
$ docker build -t myapi:1.0 -f docker/Dockerfile.prod .   # non-default file location

Instruction reference — every keyword, one line each

InstructionDoesThe thing people get wrong
FROM img [AS name]Sets the base image; starts a build stage.Pin it (python:3.12-slim, not python). FROM scratch = empty base for static binaries.
RUN cmdExecutes a command at build time; snapshots the diff.Each RUN = a layer. Chain related steps with &&; clean caches in the same RUN or the bytes stay in the layer.
COPY src dstCopies files from build context (or --from=stage) into the image.Supports --chown=user:grp and --chmod. Context-relative only — no COPY ../outside.

CMD vs ENTRYPOINT — the eternal interview question

Mental modelENTRYPOINT is the program; CMD is its default arguments. docker run image extra args replaces CMD but keeps ENTRYPOINT. Design rule: put the immutable executable in ENTRYPOINT, the overridable defaults in CMD.
Dockerfile has…docker run img executesdocker run img --debug executes
CMD ["srv","--port","80"] onlysrv --port 80--debug ← CMD fully replaced (and this fails!)
ENTRYPOINT ["srv"] onlysrvsrv --debug
ENTRYPOINT ["srv"]
CMD ["--port","80"]		srv --port 80srv --debug ← the recommended pattern
WATCH OUT

Exec form vs shell form is a signals issue, not style. CMD ["python","app.py"] (exec form, JSON array) makes python PID 1 → it receives SIGTERM on docker stop. CMD python app.py (shell form) wraps it as /bin/sh -c "python app.py" → sh is PID 1, doesn't forward signals, and your app gets SIGKILLed after the 10 s grace period. Always exec form for ENTRYPOINT/CMD; shell form is fine inside RUN.

$ docker run myimg --port 9090            # replaces CMD (appended to ENTRYPOINT)
$ docker run --entrypoint sh -it myimg    # replaces ENTRYPOINT (great for poking around)

Layer caching — why build order is the whole game

For each instruction the builder asks: same instruction text, same input files (checksummed for COPY/ADD), same parent layer? If yes → reuse the cached layer. The first miss invalidates every later step. Therefore: order lines from least-changing to most-changing.

FROM node:22-slim
WORKDIR /app
COPY . .                       # ← source changes on every commit…
RUN npm ci                     # ← …so this never hits cache. 3-minute builds forever.
CMD ["node", "server.js"]
FROM node:22-slim
WORKDIR /app
COPY package.json package-lock.json ./   # changes rarely
RUN npm ci                               # cached until the lockfile changes
COPY . .                                 # changes often — lives below the expensive step
CMD ["node", "server.js"]
$ docker build --no-cache -t app .    # ignore cache entirely (debug “stale build” suspicions)
$ docker build --pull -t app .        # also re-check the base image for updates
SENIOR LENS

This one habit compounds more than any other Docker skill. The lockfile-first pattern (requirements.txt / package-lock.json / go.mod before source) is the difference between 3-second and 3-minute inner loops, and between cheap and expensive CI. When a teammate says “Docker builds are slow”, read their Dockerfile top-to-bottom asking one question per line: how often does this line's input change? Sort descending stability, done.

Multi-stage builds — compile heavy, ship light

Mental modelA workshop and a showroom. Stage 1 gets all the messy tools (compilers, SDKs, dev deps) and produces the artifact. Stage 2 starts from a clean minimal base and copies in only the artifact. The workshop never ships.
# ---- stage 1: build ----
FROM golang:1.23 AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download                       # cached dependency layer
COPY . .
RUN CGO_ENABLED=0 go build -o /out/server ./cmd/server

# ---- stage 2: runtime ----
FROM gcr.io/distroless/static-debian12    # no shell, no package manager, ~2 MB
COPY --from=builder /out/server /server
USER nonroot
ENTRYPOINT ["/server"]
FROM node:22-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci                                # includes devDependencies (tsc etc.)
COPY . .
RUN npm run build && npm prune --omit=dev

FROM node:22-slim
ENV NODE_ENV=production
WORKDIR /app
COPY --from=build /app/node_modules ./node_modules

Running containers — docker run & the lifecycle

One command carries half of Docker's surface area. Learn its anatomy once, and every flag becomes a slot instead of trivia.

Anatomy of a docker run

docker run -d --name api --restart unless-stopped -p 8080:8000 -e APP_ENV=prod --env-file .env -v api-data:/data --memory 512m --cpus 1 myapi:1.0 --verbose

Order matters once: everything before the image name configures the container; everything after it becomes the command (replacing CMD). A flag placed after the image is silently handed to your app — the #1 “my flag does nothing” bug.

Interactive vs detached — what -it and -d really mean

Restart-policy judgment: unless-stopped for long-lived services on a single host (survives daemon reboots but respects your manual stop); on-failure:5 for jobs that may legitimately finish; never always on a crash-looping container you're debugging — it will fight you. And once you're writing three of these flags repeatedly, that's the signal to move the definition into Compose (ch 11), where it's versioned instead of living in your shell history.

The container lifecycle

$ docker create --name api -p 8080:8000 myapi:1.0   # made, not started
$ docker start api                                  # start (add -a to attach output)
$ docker stop api          # SIGTERM → wait 10 s (-t N to change) → SIGKILL
$ docker restart api       # stop + start
$ docker kill api          # SIGKILL now (-s HUP etc. to send another signal)
$ docker pause api && docker unpause api            # freeze via cgroups (SIGSTOP-like)
$ docker wait api          # block until exit; prints the exit code
$ docker rm api            # delete a stopped container (-f = kill + delete)
$ docker rename api api-old
$ docker update --memory 1g --cpus 2 api            # change limits live
WATCH OUT

stop vs kill is graceful vs murder. stop gives the app its STOPSIGNAL and a grace period to flush buffers and close connections; kill doesn't. If your app “loses data on restart”, check two suspects: shell-form CMD (PID 1 never sees the signal — ch 06) or a grace period shorter than your shutdown routine (docker stop -t 30).

Inspecting & debugging containers

Ninety percent of container debugging is four commands — ps, logs, exec, inspect — asked in the right order.

docker ps — what's running (and what died)

$ docker ps                      # running containers
$ docker ps -a                   # + exited ones ← where crashed containers hide
$ docker ps -a --filter status=exited
$ docker ps --filter name=api --filter "label=env=prod"
$ docker ps -q                   # IDs only → compose with other commands
$ docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
$ docker ps -s                   # add writable-layer disk size per container
TIP

“My container disappeared” → it exited. docker ps -a shows it with an exit code in STATUS; then docker logs it. Nothing in Docker vanishes silently until rm.

docker logs — stdout/stderr, captured

Containers don't write log files (well-behaved ones don't) — they print to stdout/stderr, and Docker's logging driver captures the stream. This is why crashed containers can still be interrogated.

$ docker logs api                       # everything so far
$ docker logs -f api                    # follow live (Ctrl-C detaches, app keeps running)
$ docker logs -f --tail 100 api         # last 100 lines, then follow
$ docker logs --since 15m api           # time window (also: --until, RFC3339 stamps)
$ docker logs -t api                    # prefix each line with a timestamp
$ docker logs api 2>&1 | grep -i error  # logs preserve the stdout/stderr split
WATCH OUT

The default json-file driver grows without limit — chatty containers fill disks. Cap it globally in /etc/docker/daemon.json (ch 13) or per-container: --log-opt max-size=10m --log-opt max-file=3.

docker exec — a side door into a running container

$ docker exec -it api sh                 # shell in (bash if the image has it)
$ docker exec api env                    # one-shot command, print, done
$ docker exec -u root -it api sh         # enter as root even if image runs non-root
$ docker exec -it -w /app api sh         # set working dir
$ docker exec -e DEBUG=1 api ./task.sh   # extra env for this exec only
execattach
Connects toA new process you specifyThe existing PID 1's stdio
Exit effectOnly your side process endsCtrl-C usually kills the app; detach with Ctrl-p Ctrl-q
Use forDebugging (almost always this)Rejoining an interactive app you ran detached
SENIOR LENS

Exec is a stethoscope, not a screwdriver. Read files, run diagnostics, test connectivity — fine. Editing code or config via exec creates an unreproducible container that dies with the next redeploy (ch 01's whole point). If an exec'd fix works, its real destination is the Dockerfile or Compose file.

docker inspect — the full truth, queryable

Everything Docker knows about an object as one JSON blob. --format takes a Go template so you can extract exactly one fact — memorize a few and you'll stop scroll-hunting:

$ docker inspect api --format '{{.State.Status}}'                    # running / exited…
$ docker inspect api --format '{{.State.ExitCode}} {{.State.OOMKilled}}'
$ docker inspect api --format '{{.RestartCount}}'                    # crash-looping?
$ docker inspect api --format '{{range .NetworkSettings.Networks}}{{.IPAddress}} {{end}}'
$ docker inspect api --format '{{json .Config.Env}}'                 # effective env
$ docker inspect api --format '{{json .Mounts}}'                     # what's mounted where
$ docker inspect api --format '{{json .HostConfig.PortBindings}}'
$ docker inspect api --format '{{json .State.Health}}'               # healthcheck history
$ docker inspect api | less        # or just read it all

Live metrics & small utilities

$ docker stats                    # live CPU / mem-vs-limit / net / pids for all
$ docker stats --no-stream api    # one sample, script-friendly
$ docker top api                  # processes inside, from the host's view
$ docker port api                 # published-port mapping for one container
$ docker diff api                 # writable-layer changes (A/C/D)
$ docker events --since 30m       # daemon event stream: starts, dies, OOMs
$ docker cp api:/app/config.yaml ./              # copy out
$ docker cp ./patch.sql api:/tmp/                # copy in (works on stopped containers too)

Debug recipes

Recipe: a container that exits immediately

$ docker ps -a --filter name=api            # 1. confirm state + exit code
$ docker logs --tail 200 api                # 2. read its last words
$ docker inspect api --format '{{.State.ExitCode}} oom={{.State.OOMKilled}}'   # 3. decode (ch 07 table)
$ docker run --rm -it --entrypoint sh myapi:1.0   # 4. bypass ENTRYPOINT, poke around
# inside: run the real command by hand and watch it fail interactively

Recipe: debugging a shell-less image (distroless/scratch)

You can't exec into what has no shell. Borrow another container's namespaces instead — a debug sidecar sees the same network and processes:

$ docker run --rm -it \
    --pid container:api \
    --network container:api \
    nicolaka/netshoot                 # a toolbox image: curl, dig, tcpdump, htop…

# inside: `ps aux` shows api's processes; `curl localhost:8000` hits api's port

Recipe: “works locally, port unreachable” triage

#   ↳ works → host-side issue: firewall, or you published a different host port

Networking

Each container gets a private network stack (ch 02's net namespace). Networking practice is deciding how those private stacks reach each other and the world.

The model: bridges, veth cables, NAT

Mental modelA network switch inside your host. A Docker bridge network is a virtual switch; each container plugs in with a virtual cable (veth pair) and gets a private IP. Containers on the same switch talk freely. -p drills a NAT hole from a host port to one container's port for the outside world.
DriverWhat it doesReach for it when
bridgePrivate switch + NAT (the default)Almost everything on one host.
hostNo isolation — container shares the host's stack; -p is meaninglessMax network performance / dynamic ports (Linux-native trick).
noneLoopback onlyBatch jobs that must not talk; security sandboxes.
overlayVirtual switch spanning multiple hostsSwarm / multi-host clustering.
macvlan/ipvlanContainer appears as a device on your physical LANLegacy appliances that need a “real” LAN IP.

The rule that matters: user-defined networks give you DNS

On the default bridge, containers can only reach each other by IP (names don't resolve; the legacy --link hack is deprecated). On any network you create, Docker runs an embedded DNS server (at 127.0.0.11) that resolves container names and aliases. This one fact is why Compose services can just call each other db, api, cache.

$ docker network create backend
$ docker run -d --name db --network backend postgres:16-alpine
$ docker run -d --name api --network backend -e DB_HOST=db myapi:1.0
$ docker exec api ping -c1 db          # “db” resolves. No IPs anywhere.
TIP

Container IPs change on every recreate — never hardcode them. Names are stable; IPs are cattle.

Publishing ports — -p in all its forms

$ docker run -d -p 8080:80 nginx              # host 8080 → container 80 (all interfaces)
$ docker run -d -p 127.0.0.1:8080:80 nginx    # localhost only — dev-safe default
$ docker run -d -p 80:80 -p 443:443 nginx     # repeat per port
$ docker run -d -p 8080:80/udp app            # protocol suffix (default tcp)
$ docker run -d -p 80 nginx                   # random host port → container 80
$ docker run -d -P nginx                      # publish ALL EXPOSEd ports to random ports
$ docker port <name>                          # see what landed where
WATCH OUT

Two classics: (1) EXPOSE publishes nothing — it's metadata; only -p/-P opens ports (ch 06). (2) Port already allocated: only one process can own a host port. Find the squatter with docker ps --format '{{.Names}} {{.Ports}}' | grep 8080. Also: on Linux, -p without an IP bypasses typical firewall expectations by publishing on all interfaces — bind 127.0.0.1 for local-only services.

SENIOR LENS

Publish the edge, network the rest. In a multi-container app only the entrypoint (reverse proxy / web) needs -p; databases and internal APIs should be reachable solely on the internal network. Every published port is attack surface and a chance for host-port collisions — the least-privilege principle, applied to sockets.

$ docker network create backend                       # bridge by default
$ docker network create --subnet 10.9.0.0/24 lab      # pick your own range
$ docker network inspect backend                      # members, subnet, gateway
$ docker network connect backend api                  # containers can join N networks
$ docker network disconnect backend api
$ docker network rm backend
$ docker network prune                                # delete all unused networks

Storage — volumes, bind mounts, tmpfs

The writable layer dies with its container (ch 02). Anything that must outlive a container — databases, uploads, caches — needs a mount.

The three mounts (and the zeroth option)

Mental modelWhere do the bytes live? Writable layer → inside the container (dies with it). Volume → in Docker's managed area on the host (survives, portable). Bind mount → an exact host folder you chose (survives, but couples you to that host's layout). tmpfs → RAM (fast, gone at stop).
Named volumeBind mounttmpfs
Lives at/var/lib/docker/volumes/… (Docker-managed)Any host path you specifyHost memory
Survives container?Yes — until volume rmYes — it's just your folderNo
Best forData: databases, uploads, cachesDev: live source code, host configsSecrets in flight, scratch space
Pre-populates from image?Yes — first use of an empty volume copies the image dir's contents inNo — the host dir shadows whatever the image had thereNo (empty)
PortabilityHigh (no host paths in your config)Low (host-specific paths)—
 
$ docker run -d -v pgdata:/var/lib/postgresql/data postgres:16      # named volume
$ docker run --rm -it -v "$PWD":/app -w /app node:22 bash           # bind mount (dev)
$ docker run -d --tmpfs /tmp:size=64m myapi:1.0                     # tmpfs
$ docker run -d -v "$PWD/nginx.conf":/etc/nginx/nginx.conf:ro nginx # single file, read-only
WATCH OUT

The pre-population difference bites hard: bind-mount an empty host dir over /var/lib/mysql and the DB starts blank; use an empty named volume and the image's seed content is copied in on first run. When “my files disappeared”, the answer is almost always “a mount is shadowing them”.

-v vs --mount — same engine, two dialects

# -v : terse.  [source]:[dest][:options]
$ docker run -v pgdata:/var/lib/postgresql/data:ro postgres:16

# --mount : explicit key=value. Verbose, self-documenting, better errors:
$ docker run --mount type=volume,src=pgdata,dst=/var/lib/postgresql/data,readonly postgres:16
$ docker run --mount type=bind,src="$PWD"/conf,dst=/etc/app,readonly myapi:1.0
$ docker run --mount type=tmpfs,dst=/scratch,tmpfs-size=64m myapi:1.0

Behavioral difference worth knowing: with a bind mount, -v /no/such/dir:/x silently creates the host directory (as root); --mount errors instead. Scripts and docs increasingly prefer --mount for that reason; muscle memory prefers -v. Both are fine — be consistent.

Volume CLI

$ docker volume create pgdata
$ docker volume ls
$ docker volume inspect pgdata          # Mountpoint = where bytes really live on host
$ docker volume rm pgdata               # refuses while a container references it
$ docker volume prune                   # remove UNUSED anonymous volumes
$ docker volume prune -a                # remove ALL unused volumes (named too) — careful

# Which container is holding a volume?
$ docker ps -a --filter volume=pgdata
DANGER

Volumes are where the only irreplaceable thing — data — lives. docker volume prune -a and docker compose down -v are the two commands that have deleted more production databases than any hacker. Before either: docker volume ls and read the names slowly.

Everyday patterns

Database with durable data

$ docker run -d --name db \
    -e POSTGRES_PASSWORD=devpass \
    -v pgdata:/var/lib/postgresql/data \
    postgres:16-alpine
# destroy & recreate the container freely — data persists in pgdata

Dev loop: edit on host, run in container

$ docker run --rm -it \
    -v "$PWD":/app \
    -v /app/node_modules \        # anonymous volume masks this subpath → container's own deps
    -w /app -p 3000:3000 node:22 npm run dev

Backup & restore a volume

$ docker run --rm -v pgdata:/data -v "$PWD":/backup alpine \
    tar czf /backup/pgdata.tar.gz -C /data .
$ docker run --rm -v pgdata:/data -v "$PWD":/backup alpine \
    tar xzf /backup/pgdata.tar.gz -C /data
SENIOR LENS

Permissions are the recurring storage bug. A mount doesn't change file ownership: if the image runs as UID 999 and your bind-mounted host dir is owned by UID 1000, writes fail with EACCES. Fixes in order of preference: run the container as your UID (-u "$(id -u):$(id -g)"), chown the host dir once, or use a named volume (Docker sets ownership on first use). On Linux this is real UIDs; Docker Desktop papers over it with a file-sharing layer — which is also why bind-mount I/O is slower on Mac/Windows.

Docker Compose — declaring the app

Once an app is two containers plus a network plus a volume, imperative docker run commands become unmanageable. Compose flips the model: declare the desired state in YAML; one command reconciles reality to match.

First, the name confusion: docker-compose vs docker compose

TIP

Practical rule: type docker compose; read docker-compose in old tutorials as the same thing. V2 reads V1 files fine — just delete the top-level version: line. Preferred filename today is compose.yaml (with compose.yml, docker-compose.yml, docker-compose.yaml also auto-detected, in that order).

Shape of a Compose file

name: shop            # project name (defaults to the directory name)

services:             # the containers — the heart of the file
  api: { … }
  db:  { … }

volumes:              # named volumes (ch 10) referenced by services
  db-data:

networks:             # user-defined networks (ch 09); optional — a default is created
  backend:

configs:              # non-secret files to inject (spec-level)
secrets:              # secret files to inject

The project name namespaces everything (containers, networks, volumes become shop-api-1, shop_db-data, …), which is what lets several Compose projects coexist on one machine.

A complete, realistic stack

Web frontend + API + Postgres + Redis, with health-gated startup, an isolated backend network, and durable data:

name: shop

services:
  web:
    build: ./web
    ports:
      - "8080:80"                      # the ONLY published port (ch 09: publish the edge)
    depends_on:
      api:
        condition: service_healthy     # wait for real readiness, not just “started”
    networks: [frontend]
    restart: unless-stopped

  api:
    build:
      context: ./api
      target: runtime                  # multi-stage target (ch 06)
    environment:
      DATABASE_URL: postgres://app:${DB_PASSWORD}@db:5432/app   # “db” = DNS name (ch 09)
      REDIS_URL: redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    healthcheck:
      test: ["CMD", "curl", "-fsS", "http://localhost:8000/health"]
      interval: 10s
      timeout: 3s
      retries: 5
      start_period: 15s                # grace period before failures count
    networks: [frontend, backend]      # the bridge between the two zones
    restart: unless-stopped
    deploy:
      resources:
        limits: { memory: 512M, cpus: "1.0" }

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: app
      POSTGRES_DB: app
      POSTGRES_PASSWORD: ${DB_PASSWORD:?set DB_PASSWORD in .env}   # fail loudly if unset
    volumes:
      - db-data:/var/lib/postgresql/data          # durable (ch 10)
      - ./initdb:/docker-entrypoint-initdb.d:ro   # seed scripts, read-only bind
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d app"]
      interval: 5s
      retries: 10
    networks: [backend]                # NOT on frontend, NOT published
    restart: unless-stopped

  cache:
    image: redis:7-alpine
    command: ["redis-server", "--appendonly", "yes"]
    volumes: [cache-data:/data]
    networks: [backend]
    restart: unless-stopped

volumes:
  db-data:
  cache-data:

networks:
  frontend:
  backend:
$ echo 'DB_PASSWORD=s3cret' > .env
$ docker compose up -d           # build what's missing, create nets/volumes, start in order
$ docker compose ps              # web :8080 up, db healthy, cache running
$ docker compose down            # stop & remove containers + networks (volumes survive)
SENIOR LENS

Why depends_on alone isn't enough: it orders container starts, but Postgres accepting connections comes seconds later — the root cause of every “connection refused on first boot” bug. The fix is the pattern above: give the dependency a healthcheck, gate on condition: service_healthy. And still make the app retry its DB connection — in production (Kubernetes, anything distributed) there is no start ordering at all; resilience belongs in the app, ordering is just a dev-time comfort.

Network zoning is free security: db here is unreachable from web and from the host — only api bridges the zones. Two lines of YAML for a real trust boundary.

Variables & the .env file

Compose interpolates ${VAR} in the YAML before doing anything. Values come from your shell environment, else from a .env file next to the compose file (shell wins).

services:
  api:
    image: myapi:${TAG:-dev}          # default if unset/empty
    environment:
      DB_PASSWORD: ${DB_PASSWORD:?must be set}   # hard error if missing
      LOG_LEVEL: ${LOG_LEVEL-info}               # default only if UNSET
WATCH OUT

Two different jobs, one filename: .env feeds the YAML's ${…} substitution; env_file: feeds the container's environment. A variable in .env does not reach your app unless the YAML passes it through (environment: DB_PASSWORD: ${DB_PASSWORD}) or you also list the file under env_file:. Verify what will really run with docker compose config.

Compose — the CLI workflow

Every command runs against the compose file in the current directory (or -f path) and scopes itself to that project — down here never touches your other stacks.

The core loop

$ docker compose up -d               # create/start everything (pull & build as needed)
$ docker compose up -d --build       # force rebuild of build: services first
$ docker compose ps                  # this project's containers + health + ports
$ docker compose logs -f api         # follow one service (omit name = all, interleaved)
$ docker compose logs -f --tail 100
$ docker compose exec api sh         # shell into a RUNNING service (no -it needed)
$ docker compose stop                # stop, keep containers (fast resume: start)
$ docker compose start
$ docker compose restart api         # bounce one service
$ docker compose down                # stop & REMOVE containers + project networks
$ docker compose down -v             # …and the project's volumes  ← data gone (ch 10)
$ docker compose down --rmi local --remove-orphans   # scorched-earth reset
WATCH OUT

up is a reconciler, not a launcher. Re-running up -d only recreates services whose config or image changed — that's the feature. Corollaries: editing environment: requires re-running up -d (not restart, which reuses the old container as-is); editing code requires up -d --build. If state seems haunted: docker compose up -d --force-recreate.

run vs exec — one-off tasks

compose execcompose run --rm
TargetThe existing running containerA fresh container of that service
Needs service up?YesNo (starts depends_on deps for you)
Portsalready publishedNOT published unless --service-ports
Use forPoking a live serviceMigrations, seeds, test suites, CLIs
 
$ docker compose run --rm api python manage.py migrate
$ docker compose run --rm api npm test
$ docker compose run --rm --no-deps api sh     # skip starting the database etc.

Multiple files, overrides, profiles

Compose merges files left-to-right: later files override scalars and extend lists. If compose.override.yaml exists it's merged automatically — the idiomatic “base = truth, override = local dev tweaks” split:

services:
  api:
    build: { target: dev }
    volumes:
      - ./api:/app                 # live-code bind mount, dev only
    environment:
      DEBUG: "1"
    ports:
      - "9229:9229"                # debugger
$ docker compose up -d                                   # base + override (auto)
$ docker compose -f compose.yaml up -d                   # base ONLY (prod-like)
$ docker compose -f compose.yaml -f compose.prod.yaml up -d   # explicit stacking
$ docker compose config                                  # print the merged result — trust this, not your memory

Profiles — optional service groups

services:
  pgadmin:
    image: dpage/pgadmin4
    profiles: [debug]        # not started by default
$ docker compose up -d                    # pgadmin stays off
$ docker compose --profile debug up -d    # now it starts too

Scaling & live-reload

$ docker compose up -d --scale worker=3
# requires: no container_name, no fixed host port on that service
# (three containers can't share one host port — use a range "8000-8002:8000" or a proxy)
services:
  api:
    build: ./api
    develop:
      watch:
        - action: sync            # copy changed files into the container
          path: ./api/src
          target: /app/src
        - action: rebuild         # rebuild image when deps change
          path: ./api/package.json
$ docker compose watch            # or: docker compose up --watch

Project control & the rest of the toolbox

$ docker compose -p demo2 up -d      # same files, separate project (parallel stacks)
$ docker compose ls                  # all compose projects on this engine
$ docker compose pull                # refresh image: services
$ docker compose build --no-cache api
$ docker compose top                 # processes per service
$ docker compose cp api:/app/report.pdf ./
$ docker compose port web 80         # which host port did web:80 get?
$ docker compose events              # project-scoped event stream
SENIOR LENS

Where Compose ends: it manages one machine. No multi-host scheduling, no self-healing beyond restart policies, no rolling deploys with health gating. That's Kubernetes/Swarm territory — but the mental model transfers cleanly: a Compose service ≈ a Deployment + Service; depends_on health gating ≈ readiness probes. Learn Compose deeply and Kubernetes becomes “Compose with a scheduler”, not a new universe. Also: Compose files are code — commit them, review them, and treat docker compose config as their compiler output.

CH 13 · Practice

Housekeeping & system management

Docker never deletes anything on its own. Left alone, images, stopped containers, build cache, and logs will eat any disk. Here's the maintenance routine.

Where the disk went

$ docker system df            # the four buckets: images, containers, volumes, build cache
TYPE            TOTAL   ACTIVE   SIZE      RECLAIMABLE
Images          24      6        9.8GB     7.2GB (73%)
Containers      11      4        420MB     390MB (92%)
Local Volumes   9       3        2.1GB     1.4GB (66%)
Build Cache     212     0        5.3GB     5.3GB

$ docker system df -v         # per-object breakdown — find the whales
$ docker container prune                       # stopped containers
$ docker image prune                           # dangling images only
$ docker image prune -a --filter "until=168h"  # unused images older than 7 days
$ docker builder prune                         # build cache
$ docker network prune
$ docker volume prune                          # unused ANONYMOUS volumes

$ docker system prune                          # stopped ctrs + dangling images + nets + cache
$ docker system prune -a --volumes             # ← everything unused, incl. ALL volumes
DANGER

docker system prune -a --volumes is the correct command exactly twice a year and a disaster the rest of the time. It keeps only what a currently existing container uses — a database that happens to be stopped-and-removed at that moment loses its volume. Run the surgical prunes first; reach for the nuclear one only when you can name what will die.

Daemon configuration — /etc/docker/daemon.json

{
  "log-driver": "json-file",
  "log-opts": { "max-size": "10m", "max-file": "3" },   # stop log files growing forever
  "default-address-pools": [
    { "base": "10.200.0.0/16", "size": 24 }             # avoid clashing with your VPN/LAN
  ]
}
$ sudo systemctl restart docker     # apply (Linux). Desktop: Settings → Docker Engine.
$ docker info                       # verify effective config, storage driver, cgroup v2

Contexts — one CLI, many daemons

$ docker context create prod --docker "host=ssh://ubuntu@prod-server"
$ docker context ls
$ docker context use prod           # every docker/compose command now targets prod
$ docker --context prod compose ps  # or per-command, without switching
$ docker context use default
TIP

Contexts over SSH are the simplest “deployment platform” that exists: docker --context prod compose up -d deploys your stack to a server with zero agents installed. Perfectly respectable for small production.

CH 14 · Practice

Security essentials

Docker's defaults are decent (ch 02); production wants deliberate. Each measure below is one flag or one Dockerfile line — the cost/benefit is absurdly good.

The hardening ladder

MeasureHowDefends against
1 · Don't run as rootDockerfile: USER app (create it first) or run-time -u 1000:1000Container escape severity, file-permission blast radius. The single highest-value line.
2 · Minimal base image-slim / alpine / distroless / scratchCVEs in packages you never needed; gives attackers no shell, no curl.
3 · Read-only rootfs--read-only --tmpfs /tmp (Compose: read_only: true)Malware persistence, config tampering. Apps needing scratch space get tmpfs.
4 · Drop capabilities--cap-drop ALL --cap-add NET_BIND_SERVICEKernel attack surface (ch 02's power slices, minimized).
5 · No privilege escalation--security-opt no-new-privileges:truesetuid-binary tricks inside the container.
6 · Resource limits--memory --cpus --pids-limitDoS / fork bombs taking the host (ch 02).
7 · Pin & scan imagesdigests for prod; docker scout cves img / trivy in CISupply-chain swaps, known CVEs shipping unnoticed.
8 · Secrets out of images & envBuildKit secret mounts (ch 06); runtime secret files; never ENV/ARGdocker history & inspect leaking credentials.
 
$ docker run -d --name api \
    --read-only --tmpfs /tmp \
    --cap-drop ALL --cap-add NET_BIND_SERVICE \
    --security-opt no-new-privileges:true \
    --memory 512m --cpus 1 --pids-limit 256 \
    -p 127.0.0.1:8080:8000 \
    myapi@sha256:9f8e7d6c…
services:
  api:
    image: myapi@sha256:9f8e7d6c…
    read_only: true
    tmpfs: [/tmp]
    cap_drop: [ALL]
    cap_add: [NET_BIND_SERVICE]
    security_opt: ["no-new-privileges:true"]
    deploy:
      resources:
        limits: { memory: 512M, cpus: "1.0" }
DANGER

Three things that hand over the host: --privileged, mounting /var/run/docker.sock into a container, and running untrusted images unsandboxed. Treat all three as “root access granted”, because that's literally what they are.

SENIOR LENS

How to think about it: assume the app will be compromised, then design the container so the attacker wakes up in an empty, read-only, unprivileged, rate-limited room with no tools and one door (the published port). Every ladder rung above removes one thing you'd otherwise read about in the incident report. Rungs 1–2 belong in every image you ever build; 3–6 in anything internet-facing.

The recipe wall — one-liners to keep

Muscle-memory commands, grouped by moment of need. Each references the chapter that explains why it works.

Daily driving

$ docker run --rm -it alpine sh
$ docker exec -it NAME sh
$ docker compose logs -f --tail 100 api
$ docker compose up -d --build api

  -v "$PWD":/usr/share/nginx/html:ro nginx

  | grep 8080

When something's wrong

$ docker ps -a --filter name=api
$ docker logs --tail 200 api
$ docker inspect api --format \
  '{{.State.ExitCode}} oom={{.State.OOMKilled}}'
$ docker run --rm -it --entrypoint sh IMAGE
$ docker run --rm -it \
  --network container:api nicolaka/netshoot
$ docker stats
$ docker inspect api --format '{{range \
  .NetworkSettings.Networks}}{{.IPAddress}}{{end}}'
$ docker cp api:/app/error.log ./

Cleanup (ascending destructiveness)

$ docker container prune
$ docker image prune
$ docker rm -f $(docker ps -aq)
$ docker builder prune
$ docker system prune -a --volumes

Shipping

$ docker build -t ghcr.io/acme/app:1.0 .
$ docker push ghcr.io/acme/app:1.0
$ docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t ghcr.io/acme/app:1.0 --push .
$ docker save app:1.0 | gzip > app.tgz
$ gunzip -c app.tgz | docker load
$ docker --context prod compose up -d
