03The four clone attempts

All attempts used rpi-clone (partition → mkfs.ext4 → rsync ~67 GB). Each retry removed another variable. Each failed anyway.

Attempt A FAIL @ ~45 GB ext4 forced read-only mid-rsync

df froze at 45 G for ~30 min with zero live iostat activity. dmesg held the truth:

blk_update_request: I/O error, dev sda ... op WRITE
EXT4-fs (sda2): I/O error while writing superblock
EXT4-fs (sda2): Remounting filesystem read-only
EXT4-fs error: comm rsync: Detected aborted journal

Side effect: a reboot accidentally booted the incomplete SSD clone, because rpi-clone labeled the SSD root writable — colliding with the SD card's label while cmdline.txt used root=LABEL=writable. Recovered over SSH by pinning root=PARTUUID=b45a3eea-02 and relabeling the SSD. Also produced the Docker readlink overlay2 error — an overlay layer that was never fully copied.

Attempt B FAIL — read-back corruption after fio raw-device damage + forced clone

EXT4-fs error ... error -5 reading directory block — freshly written blocks came back unreadable. Confounded by two self-inflicted wounds: fio --filename=/dev/sda had wiped the partition table with raw writes, and the clone was forced over that mess with rpi-clone -f.

Attempt C FAIL @ mkfs clean wipe · UAS quirk active · no -f

Disk wiped properly (wipefs + sgdisk --zap-all), UAS disabled via usb-storage.quirks=0bda:9201:u (confirmed active; throughput dropped ~3× as expected). Died before rsync even started:

usb 2-2: reset SuperSpeed USB device ... (×6)

Attempt D FAIL — bus departure USB 3.0 port · solo device · Docker stopped · power verified 0x0

The controlled final run: only USB device, blue port, Docker + containerd stopped, quirk confirmed (UAS is ignored ... using usb-storage instead), power clean. The bridge reset 24 seconds after enumerating, flooded ext4 with error -5 read-back failures, then left the bus entirely — sda vanished from lsblk while rsync sprayed mkdir ... 05Final forensic sequence (attempt D)

  1. Enumeration warnings: U1/U2 device-initiated enable failed — the bridge can't negotiate basic USB 3.0 link power states. Firmware broken at the protocol level.
  2. +24 s under write load: first reset SuperSpeed USB device.
  3. +33 s: ext4 read-back corruption — error -5 reading directory block on blocks rsync had just written.
  4. Terminal event: full USB disconnect; sda absent from lsblk; rsync generator fails every remaining mkdir with EIO; rpi-clone aborts at mkfs/mount of sda1.
  5. Prior history: error -71 (EPROTO) enumeration failures logged long before the clone — pre-existing defect, not load-induced degradation alone.

Root cause

The Realtek RTL9201 USB-SATA bridge resets and disconnects under sustained write load, and shows USB protocol errors even at idle enumeration. No software setting fixes hardware that physically leaves the bus. Fix: replace the adapter (ASMedia ASM1153E/ASM1351 or JMicron recommended; avoid Realtek).

06Key technical lessons

btop "Sleeping" vs "Waiting": S = idle/normal (tmux server lives here forever — red herring). D = uninterruptible I/O sleep — means "busy writing" only if the device is healthy; on a dead device it means "blocked forever." Always cross-check dmesg.
High load average + low CPU% = I/O-bound. The normal signature of a disk-to-disk copy, not a hung system.
rsync forks into three PIDs (generator / sender / receiver) — three rsync processes is normal, not a runaway.
iostat -x N N: block 1 = since-boot averages (polluted by e.g. the one-time mkfs TRIM), block 2 = the live window. Trust block 2 for "right now."
Flat df + zero live iostat for minutes ≠ "slow small-files phase." Check dmesg for device errors immediately — don't wait 30 minutes.
Never run fio --filename=/dev/sdX against a real device — raw writes destroy the partition table. Use a file path on a mounted filesystem or read-only mode.
Never use rpi-clone -f to bulldoze past "unrecognised disk label." Wipe properly first: wipefs -a → sgdisk --zap-all → partprobe.
Label collisions boot the wrong disk. Two partitions labeled writable + root=LABEL=writable = firmware picks whichever it likes. Pin root by PARTUUID.
Stop Docker before cloning a live system (docker docker.socket containerd) — overlay2's thousands of small files and layer symlinks clone half-consistent while containers write.
vcgencmd "VCHI initialization failed" = permissions (needs sudo or the video group), not a real failure. Fallback: kernel hwmon rpi_volt or dmesg | grep -i voltage.
Isolating a bridge fault: SMART clean (media + CRC) + fio clean + power 0x0 + failures persisting across UAS quirk, port, and load changes = the USB-SATA bridge, by elimination. USB reset/disconnect/error -71 lines are its fingerprints.
UAS quirk tradeoff: usb-storage.quirks=VID:PID:u serializes I/O — expect ~3× slower (observed: read 40→13 MiB/s, write 17→5.7 MiB/s). Fine for a one-time clone; good bridges don't need it.
USB 2.0 vs 3.0 on Pi 4: real-world ~30–40 MB/s vs ~200–350 MB/s. Always use the blue ports for storage; both 3.0 ports share one controller.

07Command cookbook

# Identify disks, labels, UUIDs, PARTUUIDs
lsblk
sudo blkid
# SMART health through a USB bridge
sudo smartctl -d sat -a /dev/sda | grep -iE 'health|realloc|pending|error'
# Properly wipe a disk to factory-blank (NOT fio, NOT rpi-clone -f)
sudo wipefs -a /dev/sda
sudo sgdisk --zap-all /dev/sda
sudo partprobe /dev/sda
# Stop Docker before cloning a live system (and restart after)
sudo systemctl stop docker docker.socket containerd
# ... clone ...
sudo systemctl start docker
# tmux pane 1 — the clone (no -f)
sudo rpi-clone sda
# tmux pane 2 — error watchdog (must stay SILENT; catches bridge + power)
sudo dmesg -w | grep -iE 'sda|I/O error|EXT4|reset|disconnect|voltage'
# tmux pane 3 — progress + power
watch -n5 'df -h /mnt/clone 2>/dev/null; echo; vcgencmd get_throttled'
# Power / undervoltage checks (0x0 = clean; 0x50005 = undervoltage NOW)
sudo vcgencmd get_throttled
dmesg | grep -iE 'voltage|throttl'
cat /sys/class/hwmon/hwmon*/name           # look for rpi_volt
cat /sys/class/hwmon/hwmon*/in0_lcrit_alarm  # 1 = undervoltage now
# Confirm UAS quirk applied (Realtek bridge only)
cat /sys/module/usb_storage/parameters/quirks   # expect 0bda:9201:u
lsusb -t                                         # expect Driver=usb-storage
# Verify which partition firmware boots / where root is
findmnt /boot/firmware   # mmcblk0p1 = SD boot
findmnt /                # mmcblk0p2 = running on SD
# Live disk throughput
iostat -x 2 2 | grep -E "Device|sda|mmcblk"   # 2nd block = live window
iostat -x 0.1 | grep --line-buffered -E "Device|sda|mmcblk"

08Current system state

The SD-card system was never at risk

  • Booted and running on mmcblk0p2 — healthy, stable, in normal use.
  • Boot pinned to the SD card: cmdline.txt uses root=PARTUUID=b45a3eea-02 (immune to label collisions).
  • SSD root was relabeled ssd-clone-bad during recovery; SSD now holds only a garbage half-clone / partial state — disposable.
  • UAS quirk usb-storage.quirks=0bda:9201:u remains in cmdline.txt — harmless; keyed only to the Realtek's USB ID, ignored by any other adapter.
  • Docker may need restarting after the last attempt: sudo systemctl start docker.

09Next steps — with the replacement adapter

  1. Buy an ASMedia (ASM1153E / ASM1351) or JMicron USB-SATA adapter. Avoid Realtek. External power / powered enclosure is nice-to-have, not required — power was exonerated on this Pi.
  2. Plug in SSD via the new adapter → lsusb -t: expect a new USB ID and Driver=uas (good bridges don't need the quirk; the old quirk won't match and can stay).
  3. Wipe the SSD blank: wipefs -a → sgdisk --zap-all → partprobe.
  4. Stop Docker: sudo systemctl stop docker docker.socket containerd.
  5. Run the 3-pane tmux setup (clone / dmesg watchdog / df+power watch) → sudo rpi-clone sda, no -f.
  6. Success criteria: df climbs past the old 45 GB failure point to ~67 GB with a silent watchdog. With UAS working, expect minutes, not hours.
  7. Before switching boot to the SSD: verify labels with sudo blkid; keep root pinning unambiguous (PARTUUID) on whichever disk boots; restart Docker.
